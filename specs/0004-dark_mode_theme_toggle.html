Date: 2026-07-17

The user-facing "Naziv polja" field-label mapping the spec-planner normally settles does not apply here — this feature adds no data fields; its only user-facing text is the toggle's accessible label and title (§7).

1. Summary

Add a light / dark theme toggle to the app. A single control in the top-right of the page header, opposite the <h1> title, switches the whole app between a light theme and a dark theme. The control shows a sun or moon icon reflecting the active theme and is visible at all times, on every tab. On first visit the app follows the operating system's colour-scheme preference; once the user makes an explicit choice, that choice is remembered locally (browser storage, same privacy model as spec 0002) and restored on the next visit. Purely a UI/UX change; nothing leaves the browser.

2. Background / motivation

The app currently renders a single fixed light theme. A dark theme reduces eye strain in low-light conditions, matches user and OS-level expectations (most platforms now expose a system-wide light/dark preference), and is a low-risk, self-contained enhancement to an otherwise data-entry-heavy tool people may use for extended sessions. Because the app is fully client-side with no backend, the preference — like all app state — must be handled locally and never leave the browser.

3. Scope

In scope

  • A single theme-toggle control placed in the top-right of the page header, horizontally opposite the <h1> "DDV Poročilo" title (§4).
  • The control is always visible, independent of the active tab ("Poročilo" or "Pomoč") — it lives in the header, outside both tab panels (§4).
  • The control's icon reflects the active theme: a sun icon in light mode, a moon icon in dark mode; the icon swaps as the user toggles (§4, §7, decision in §11.1).
  • A complete dark theme covering every user-facing surface of the app so all content stays legible with adequate contrast when dark mode is active (§6).
  • First-visit default follows the OS colour-scheme preference (prefers-color-scheme); light is the ultimate fallback when no OS preference is expressed (§5, decision in §11.2).
  • Remembering an explicit choice locally: once the user toggles, that explicit theme is stored in the browser and restored on the next visit, overriding the OS default until the user toggles again (§5, decision in §11.3). Uses the same local-only, failure-tolerant storage model as spec 0002.
  • Accessibility: the icon-only control has a Slovenian accessible name, its state is exposed to assistive technology, and it is keyboard-operable (§8).
  • Slovenian for all user-facing text on the control (accessible label, optional tooltip) (§7).

Out of scope

  • More than two themes or user-customisable colours / accent picking. Exactly two themes: light and dark.
  • Per-tab or per-section theming. The theme is global to the whole app.
  • Live following of OS changes after an explicit choice. Once the user has chosen, the app honours the stored choice; it does not switch out from under the user when the OS flips (see §11.3 for the pre-choice behavior, which is the only place OS preference is consulted).
  • Syncing the preference across browsers, devices, or profiles (impossible without a backend; contrary to the app's privacy model).
  • Any change to the data model, FURS fields, validation rules, serialization, or the DDV_KIR_KPR output (specs 0001–0003 unchanged).
  • A theme setting inside the "Pomoč" page, a settings panel, or any second entry point to the toggle — there is exactly one control (§4).

4. Placement & visibility

  • Header row. The existing page <h1> "DDV Poročilo" and the new toggle sit on a shared header line: the title on the left, the toggle on the right edge of the content column, vertically aligned with the title. (The app content is centred in a fixed-max-width column; "top-right of the page" means the right edge of that column, mirroring the title's left position.)
  • Above the tab bar. The header (title + toggle) stays above the tab bar, which stays above the tab panels. The toggle is not inside either tab panel, so switching between "Poročilo" and "Pomoč" never hides or moves it.
  • Always visible. The toggle is shown on every tab and in every state of the app (empty form, filled form, errors revealed, help open). It is never conditionally hidden.
  • Single control. There is exactly one toggle in the app. Clicking / activating it flips the theme (light → dark or dark → light).
  • Icon reflects state. The control renders a sun icon while the light theme is active and a moon icon while the dark theme is active; the icon swaps immediately on toggle (§11.1 records this convention and its rationale).
  • Self-contained icons. The sun/moon indication must be rendered without external assets (no image files, icon fonts, or third-party frameworks), consistent with the app's no-dependency, no-build architecture. Exact rendering (inline markup vs. text glyph vs. inline vector) is an implementation detail.
  • Responsive. At narrow / mobile widths the title and toggle must remain on a usable header without the toggle overflowing off-screen or overlapping the title; both stay reachable. Exact responsive treatment (wrap, shrink) is left to implementation.
  • No layout shift on toggle. Switching theme must not reflow or resize page content beyond the colour change itself, and must not scroll the page or move focus.

5. Behavior — default, toggle, persistence

  • Resolving the theme on load (in priority order):
    1. If an explicit stored preference exists (from a previous toggle), use it.
    2. Otherwise, follow the OS colour-scheme preference (prefers-color-scheme: dark → dark; light or none → light).
  • No flash of the wrong theme. The resolved theme must be applied before the app's content is visibly painted, so a returning dark-mode user does not see a light flash on load (and vice versa). This is a user-experience requirement; the mechanism is an implementation detail.
  • Toggling. Activating the control switches to the other theme immediately and applies it to the entire app. The toggle is a pure view concern: it must not alter entered Glava/KIR/KPR data, validation state (including whether errors are revealed), the active tab, scroll position, or the persisted Glava data (spec 0002). Theme and app data are independent.
  • Remembering the choice. When the user toggles, the resulting explicit choice (light or dark) is written to browser-local storage, under its own key in the app's existing namespaced, versioned, failure-tolerant storage model (spec 0002 / storage.js) — a dedicated theme key, independent of and not co-mingled with the Glava record.
  • Restoring the choice. On the next visit in the same browser on the same device, the stored explicit choice is restored automatically per the load-resolution order above.
  • Storage-failure tolerance. If reading or writing the preference fails (private/incognito restrictions, disabled storage, quota), the toggle must still work for the current session — the theme changes visibly; it just may not be remembered next time. No error is surfaced to the user for a failed remember (consistent with the app's quiet, failure-tolerant storage behavior in spec 0002).
  • Scope of a stored preference. The remembered theme is local to that browser/device only; it is never synced across browsers, devices, or profiles, and may be lost if the user clears browser data — same caveats as the Glava persistence in spec 0002.

6. Theming coverage & contrast

The dark theme must cover every user-facing surface so nothing becomes illegible or invisible when it is active. Today the app's colours come from CSS custom properties (--fg, --muted, --accent, --border, --bg-soft, --error, --success) plus a few fixed light values (white input / card / button backgrounds). The dark theme is responsible for all of the following surfaces:

SurfaceRequirement in dark mode
Page background & body textDark background, light foreground text, meeting contrast targets below.
Headings, section titles, intro / help prose, muted textLegible against the dark background; muted text still visibly de-emphasised but readable.
Tab bar & selected-tab indication (spec 0003)Selected tab stays visibly distinct and not by colour alone (weight + underline preserved); both tabs readable.
Sections, entry cards, inputs, selects, checkboxesNo white-on-white / light boxes; field text, borders, and placeholders remain legible; focus outlines remain visible.
Buttons (Prenesi JSON, Dodaj…, Zapomni si / Pozabi, Odstrani)Readable labels and borders; disabled state still distinguishable from enabled.
Validation error text & error field borders (--error)Errors remain clearly signalled and readable; red keeps its error meaning but at a dark-mode-appropriate shade with sufficient contrast.
Persist success / error feedback (--success / --error, incl. the ✓ / ✕ icons)Success (green) and error (red) stay distinguishable and legible; their semantics are unchanged.
Links (e.g. GitHub link on "Pomoč")Recognisable as links and readable against the dark background.
The theme toggle itselfSun/moon icon clearly visible in both themes.
  • Contrast target. Text and meaningful UI in both themes should meet WCAG 2.1 AA contrast (4.5:1 for normal text, 3:1 for large text and meaningful UI boundaries / focus indicators). Dark mode must not be a naive colour inversion that leaves any text or control below this bar.
  • Meaning preserved. Colour semantics carry over: error = red-family, success = green-family, accent = the app's primary accent. Dark mode adjusts shade/lightness, not meaning.
  • Reduced motion. Any transition applied to the theme change must respect prefers-reduced-motion (the app already honours this for tab transitions).

7. User-facing text (Slovenian)

All user-facing text is Slovenian, per project convention (CLAUDE.md) and specs 0001–0003. The toggle is icon-only, so its only text is its accessible name (and an optional matching tooltip / title). The accessible name must describe the action (what activating it does), because the icon alone is ambiguous; it must reflect the current state so the action is unambiguous.

Active themeIcon shownAccessible name / tooltip (Slovenian) — fixed
Light (svetli način)SunPreklopi na temni način
Dark (temni način)MoonPreklopi na svetli način
  • The icon reflects the current theme (sun = light now, moon = dark now); the accessible name states the destination action (switch to the other theme). Together these are unambiguous regardless of a viewer's icon-reading habits (see §11.1).
  • Slovenian terms used: "temni način" (dark mode) and "svetli način" (light mode); "Preklopi na …" = "Switch to …".
  • The purely decorative icon glyph must not be announced separately by screen readers (it is hidden from the accessibility tree); the button's accessible name is the sole announced text (§8).

8. Accessibility

  • Named control. The icon-only toggle exposes a Slovenian accessible name per §7, so screen readers announce a meaningful action rather than an unnamed button.
  • State conveyed. The current theme / toggle state is perceivable to assistive technology (e.g. via a pressed/toggle state and/or a state-reflecting accessible name), so a non-sighted user can tell which theme is active. Exact ARIA mechanics are an implementation/accessibility detail.
  • Keyboard operable. The toggle must be reachable in the tab order and operable by keyboard (activatable with the standard button keys), not pointer-only. It must show a visible focus indicator in both themes.
  • Decorative icon. The sun/moon glyph is hidden from the accessibility tree so it is not announced redundantly or confusingly alongside the accessible name.
  • Contrast. The toggle and its focus indicator meet the §6 contrast targets in both themes.

9. Edge cases

CaseExpected handling
First visit, OS set to dark App opens in dark mode (OS preference), no stored choice needed. Toggle shows the moon icon.
First visit, no OS preference expressed App opens in light mode (the fallback). Toggle shows the sun icon.
Returning visit with a stored explicit choice that differs from the OS Stored explicit choice wins; the app opens in the remembered theme regardless of the current OS setting.
Storage unavailable (private/incognito, disabled, quota) Toggle still switches the theme for the session; the choice simply is not remembered next time. No error shown to the user.
OS colour scheme changes while the app is open, before any explicit choice Not required to switch live; consulting the OS preference at load time satisfies the requirement. (Following it live is allowed but optional and must never override an explicit stored choice.)
Toggle activated mid data-entry (fields filled, errors revealed) Only colours change. Entered values, validation state, revealed errors, active tab, and scroll position are all preserved (§5).
Reload after toggling The app reopens in the last explicitly chosen theme (restored from local storage).
User clears browser data The stored theme choice is lost; the app falls back to the OS-preference default on the next visit — same caveat class as the Glava persistence (spec 0002).
Printing Out of scope as a feature, but the theme must not make printed output illegible; if this proves a concern it is deferred (§8/§11 note), not solved here.

11. Product decisions

The user's request left several product choices open ("clarify/decide as part of the spec"). The following were proposed using the app's existing conventions and common platform behavior, and are confirmed:

11.1 Icon convention — icon reflects the current theme

Sun icon while light is active, moon icon while dark is active (the icon depicts the current state), with the accessible name stating the action ("Preklopi na temni/svetli način"). This matches the leading example in the request ("sun icon when in light mode") and, paired with an action-worded accessible name, removes the ambiguity of the alternative "show the destination icon" convention.

11.2 First-visit default — follow the OS preference, light as fallback

Respect prefers-color-scheme on first visit; light when the OS expresses no preference. Matches user expectation and is the standard, low-surprise default.

11.3 Persistence & OS interaction — remember explicit choice locally; it overrides OS until re-toggled

The OS preference is consulted only until the user makes an explicit choice; after that, the stored explicit choice wins on every load and the app does not switch out from under the user when the OS flips. An explicit user action should be sticky, consistent with how spec 0002 treats explicitly saved data.

11.4 Deferred

  • Print behavior (§9) is out of scope for this iteration — the app is not currently optimized for printing, and dark mode does not need to solve that first.
  • Exact accent shade in dark mode. §6 fixes the requirement (AA contrast, preserved meaning) but not the specific dark-theme palette values — those are a visual-design detail left to implementation planning, within the §6 constraints.
