1. Summary

Let the user persist the general-data section (Glava) only — taxpayer number and reporting period selection plus the two header flags — in the browser's local storage, via two explicit actions: "Zapomni si" (save) and "Pozabi" (forget). When saved Glava data exists, it is auto-restored into the form the next time the app is opened. This removes the friction of re-entering the same taxpayer/period header every session. Fully client-side; nothing leaves the browser.

2. Background / motivation

The Glava header (taxpayer number, reporting period, and the VRACILO / ODBDELEZ flags — spec 0001 §5) is largely stable for a given VAT payer: the taxpayer number never changes, and the period/flags change predictably between filings. Re-typing this on every visit is repetitive friction, especially since the base application intentionally has no session persistence (spec 0001 §9). This feature adds an opt-in, local-only way to remember the Glava section so it is prefilled on the next visit, while keeping the "nothing leaves the browser" guarantee intact.

3. Scope

In scope

  • A "Zapomni si" action that saves the current Glava-section values to local storage (§5, §6).
  • A "Pozabi" action that removes the saved Glava-section values from local storage (§5, §6).
  • Auto-restore on open: if saved Glava data exists in local storage when the app loads, the form is prefilled from it automatically — no separate "load" action (§6).
  • Disabled "Pozabi" when nothing is saved, including a visually disabled state (§6).
  • Green-checkmark confirmation next to the just-clicked button after a successful save or a successful remove, visible for 3 seconds, then hidden (§6).
  • Red failure indicator next to the just-clicked button when a save or remove fails, visible for 3 seconds, then hidden (§6).
  • A forward-compatibility constraint binding the storage design so that later features can persist additional, independent data domains alongside Glava (§7).

Out of scope

  • Persisting KIR / KPR entries — this feature is Glava-only. The charged/deducted ledger lists are never saved or restored here.
  • Any server-side or account-based persistence, and any cross-device or cross-browser sync — storage is confined to the single browser's local storage.
  • Export / import of the saved data as a file, or any user-visible representation of the stored blob.
  • Encryption of the stored data, multiple named/saved profiles, and any undo of "Pozabi".
  • Any change to the DDV_KIR_KPR JSON output, the download flow, or the FURS validation rules (spec 0001 §7, §8).

4. Persisted data set

Exactly the six Glava fields are persisted — the full contents of the general section as defined in spec 0001 §5. Nothing else. The period type/unit/year are the application's own input model for choosing the period; they are not FURS wire fields themselves (spec 0001 §5 derives OBDOBJE_OD / OBDOBJE_DO / OBDOBJE from them). Only TaxPayerID, VRACILO, and ODBDELEZ are FURS wire identifiers.

5.1 Placement (proposed)

The two buttons sit in a new action row below the Glava field grid, separated from the fields by a hairline rule consistent with the existing .entry-card-header rhythm. The buttons reuse the application's existing button visual language verbatim — the established button tokens/classes and the --accent / --border / --bg-soft palette (spec 0001 styling; project CSS). The success and failure indicators reuse the existing status palette — the success check in a green consistent with the app, the failure icon in a red consistent with the existing --error token. No new visual language or bespoke component styles are introduced. This placement/layout is a confirmed-reasonable proposal derived from the design mockup; the fixed requirements below are not negotiable, but exact spacing/order within the row may be refined during implementation planning without another spec revision.

Fixed requirements (non-negotiable): (1) the two button labels are exactly "Zapomni si" and "Pozabi"; (2) "Pozabi" is disabled — including visually — whenever no Glava data is currently saved; (3) after a successful save or remove, a green checkmark appears next to the button just clicked, stays visible for 3 seconds, then disappears; (4) after a failed save or remove, a red circle-with-X icon plus the text "Poskusite znova." appears next to the button just clicked, stays visible for 3 seconds, then disappears.

6. Behavior

6.1 Saving — "Zapomni si"

  • Clicking "Zapomni si" writes the current values of all six Glava fields (§4) to local storage as a single Glava record, overwriting any previously saved Glava record.
  • Save-as-is, no validation gate: the current Glava values are saved regardless of whether they are valid or complete (e.g. an empty tax number, or no period unit chosen, is still saved). Saving is a convenience snapshot, not a submission, so it is never blocked by validation.
  • On success, a green checkmark appears next to the "Zapomni si" button for 3 seconds, then disappears.
  • Because a save means data now exists in storage, the "Pozabi" button becomes enabled immediately after a successful save (if it was disabled).
  • On failure (storage unavailable, quota exceeded, private-mode restrictions), the save does not happen: the red failure indicator is shown instead (§6.6), no data is written, and the "Pozabi" enabled/disabled state is unchanged.

6.2 Forgetting — "Pozabi"

  • Clicking "Pozabi" removes the saved Glava record from local storage.
  • It does not clear or alter the values currently shown in the form; it only deletes what is stored. (There is no undo — §3 Out of scope.)
  • On success, a green checkmark appears next to the "Pozabi" button for 3 seconds, then disappears.
  • Because storage is now empty, the "Pozabi" button becomes disabled (including visually) immediately after a successful remove.
  • On failure (the removal could not be completed), the red failure indicator is shown instead (§6.6), the stored record is treated as still present, and "Pozabi" remains enabled.

6.3 "Pozabi" enabled/disabled state

  • "Pozabi" is enabled only when a saved Glava record currently exists in local storage; otherwise it is disabled, both functionally and visually.
  • This state is evaluated on app open and updated after every successful save (→ enabled) and successful remove (→ disabled). A failed save or remove does not change this state.

6.4 Auto-restore on open

  • When the app loads, if a saved Glava record exists, the general-section form is prefilled from it. If none exists, the form uses its normal defaults (spec 0001; app default general state).
  • Auto-restore affects the Glava section only; KIR/KPR lists always start empty.
  • Restored fields show validation errors immediately. Auto-restored Glava fields are treated as already "touched", so any values that fail validation surface their error on open — a deliberate deviation from the app's normal "quiet until touched" rule (spec 0001 §7; project validation UX), which applies to fresh user-typed input.
    Rationale: a restored value can be stale or invalid (an old save, or a value that no longer passes validation after a rule/schema change). Surfacing that immediately — rather than hiding it until the user happens to touch that field — makes the problem visible and fixable at once. Validation itself is unchanged: the existing single source of truth for field errors still decides what is valid; only the visibility of those errors on restored fields differs.

6.6 Feedback indicators (success & failure)

  • Both indicators occupy the same slot / placement next to the button that was just clicked (§6.5), and share the same 3-second transient behavior — appear on the action, auto-hide after 3 seconds.
  • Success: a green checkmark icon, shown after a successful save or a successful remove.
  • Failure: a red circle-with-X icon paired with the text "Poskusite znova.", shown after a failed save or a failed remove.
  • Only one indicator is shown at a time per button, reflecting the most recent action. If a button is clicked again before its 3 seconds elapse, the timer resets and the indicator reflects the latest outcome (proposed handling; not a fixed requirement).
  • Each indicator carries an accessible text equivalent in Slovenian so meaning is not conveyed by color or icon alone — proposed wording: "Shranjeno" (successful save), "Pozabljeno" (successful remove); the failure indicator's "Poskusite znova." already serves as its text. Final accessible wording may be adjusted in implementation planning.

7. Forward-compatibility constraint

Product-level requirement (binding on implementation planning). The user has flagged that future features may persist additional, independent kinds of data to local storage. Glava is the first of potentially several persisted data domains. The persistence scheme therefore must be structured so that additional domains can be added alongside the Glava record without migrating, reshaping, or overwriting existing stored data. Persisting Glava as a single flat blob that a later feature would have to break apart or migrate away from is not acceptable.

This is a scope/forward-compatibility constraint, not an architecture decision. The concrete storage key scheme, record shape, and versioning approach are deliberately left to implementation planning (impl-planner) — this spec only requires that whatever scheme is chosen keeps distinct data domains independently addressable and independently removable.

8. Edge cases

CaseExpected handling
Local storage unavailable or writes rejected (private-mode restrictions, quota exceeded, storage disabled) The action fails safely: the app keeps working, form data in memory is untouched, no data is written/removed, and the red failure indicator ("Poskusite znova.", §6.6) is shown instead of the success checkmark. The "Pozabi" enabled/disabled state is left unchanged.
Saved Glava record is missing, empty, corrupt, or unparseable on open Treat as "nothing saved": the form uses normal defaults, and "Pozabi" is disabled. Auto-restore must never break app startup.
Saved record was written by a different/older app version and some fields are missing or unknown Restore whatever recognized Glava fields are present; fall back to defaults for any missing field. Ignore unknown fields. A restored-but-invalid value still surfaces its error immediately (§6.4).
Restored values fail validation (stale save, changed rules) Prefill them anyway and show their errors immediately (§6.4). The user corrects and may re-save.
"Pozabi" clicked while nothing is saved Not reachable through the UI — the button is disabled in that state (§6.3). If it were somehow triggered, it is a harmless no-op.
"Zapomni si" clicked repeatedly Each click overwrites the prior saved Glava record with the current values; the indicator timer resets to the latest action.

9. Non-goals (explicitly deferred)

  • Persisting KIR / KPR entries or any data beyond the six Glava fields.
  • Cross-device / cross-browser sync, cloud/account storage, or any server component.
  • Encryption of stored data.
  • Multiple named/saved Glava profiles or a history of saves.
  • Undo of "Pozabi".
  • Export / import of the saved data as a file.
  • Any change to the DDV_KIR_KPR output, the download gate, or FURS validation rules.

10. References / notes

This feature is local-storage persistence and UI behavior only. It has no dependency on any FURS form, the ZDDV-1 / ZDDV-1O statute, the Pravilnik o izvajanju ZDDV-1, or any EU VAT rule, and it does not change the DDV_KIR_KPR file, its fields, or its validation. Accordingly, no external legal or regulatory sources apply to the behavior specified here, and none are cited — a deliberate note, not an omission.

The only relevant references are internal:

  • Spec 0001 — VAT records KIR/KPR JSON export (specs/0001-vat_records_kir_kpr_json_export.html): defines the Glava fields (§5), their Slovenian "Naziv polja" labels, and the validation model / quiet-until-touched UX (§7) that this feature builds on and selectively deviates from (§6.4).
  • Project conventions (repo-root CLAUDE.md): all user-facing text is Slovenian; FURS wire-format identifiers (TaxPayerID, VRACILO, ODBDELEZ) stay verbatim; validation remains the single source of truth for field errors.
