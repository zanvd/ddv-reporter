Changelog: v1.2 — finalized user-facing field labels in Slovenian using FURS's official "Naziv polja" terminology (§5, §6); aligned the ODBDELEZ label to the DDV-O form's wording ("Izračunavam odbitni delež", §5); added requirement that all user-facing text is Slovenian (§3). v1.1 — VRACILO (VAT-refund request) moved into scope for v1 (§3, §5, §7); removed from Non-goals (§9).

1. Summary

A standalone, client-side web application that lets a Slovenian VAT payer enter their VAT ledger data through a form with descriptive (human-readable) labels, and download a correctly structured JSON file conforming to FURS's KIR/KPR VAT-records specification. No submission, accounts, or persistence — the single deliverable action is producing a downloadable JSON file.

2. Background / motivation

Since 1 July 2025 (ZDDV-1O amendment to the Slovenian VAT Act, ZDDV-1), every person identified for VAT in Slovenia must, for each tax period, submit to FURS two prescribed VAT records in addition to the DDV-O return:

  • Evidenca obračunanega DDV — the record of charged / output VAT, kept as the book of issued invoices, FURS code KIR (Knjiga izdanih računov).
  • Evidenca odbitka DDV — the record of deducted / input VAT, kept as the book of received invoices, FURS code KPR (Knjiga prejetih računov).

FURS accepts these records in XML, JSON, or CSV. Both records are delivered together in a single file built around the DDV_KIR_KPR structure, which contains a header plus two arrays of line entries (one per invoice/document). This multi-entry, charged-vs-deducted, JSON-capable format is exactly what this application targets.

Important distinction: This is not the DDV-O VAT return. DDV-O is a summary form whose eDavki import format is XML only and which has no per-line/multi-entry JSON representation. The KIR/KPR records specified here are what FURS then uses to pre-fill the DDV-O. This application produces the KIR/KPR records file, nothing else.

3. Scope

In scope (v1)

  • Single-page form to capture:
    • Header (taxpayer, reporting period, refund-request flag, deductible-proportion flag).
    • Any number of charged (KIR / output) entries.
    • Any number of deducted (KPR / input) entries.
  • Add / edit / remove entries in each list.
  • Basic client-side field-format validation (see §7).
  • A Download button that assembles a valid DDV_KIR_KPR JSON document and offers it as a file download.
  • All user-facing text is in Slovenian — every string the user sees (field labels, validation / error messages, buttons, hints, empty states) is Slovenian, using FURS's official field terminology ("Naziv polja") for the ledger fields. The FURS field codes (P2, P14, …) are internal to the JSON only and are never shown to the user.

Out of scope (v1) — see §9 for the full non-goals list

  • The DDV-O return itself, and any XML or CSV output.
  • Direct submission to eDavki (REST/SOAP web service or ZIP upload).
  • Self-disclosure (samoprijava) and other non-standard treatment cases.
  • Foreign-taxpayer / tax-representative filing.
  • User accounts, authentication, and data persistence between sessions.

4. Target output format

The downloaded file is a single JSON document with this shape (field codes are FURS's; see the mapping in §6): 

DDV_KIR_KPR
├── Glava                 (header — required)
├── Lista_KIR             (charged / output records)
│   └── KIR: [ entry, entry, … ]
└── Lista_KPR             (deducted / input records)
    └── KPR: [ entry, entry, … ]

The "multiple entries for charged and deducted VAT" requirement is realized by the KIR and KPR arrays. Each array item is one ledger line (one invoice/document).

Records with no entries: if the user enters only charged (or only deducted) records, the file still validly carries the other list as empty/absent, and the corresponding header flag is set accordingly (see KIR / KPR flags in §5).

6. Entry requirements

Both entry types share two auto-managed fields:

  • ZAPST (sequential number) — assigned automatically by the app, starting at 1 and strictly consecutive within each list (FURS rejects non-consecutive numbering).
  • OBDOBJE (tax period, MMMM format) — derived from the header period, identical for every entry in the file. MMMM = two-digit start month + two-digit end month (e.g. monthly June → 0606; Q2 → 0406).

Every entry also carries OBRAVNAVA = "1" (normal return) — hardcoded in v1. Self-disclosure values ("2", "3") and their companion fields (OBDOBJE88, DAVEK88) are out of scope.

Note on P7 (supplier country): The official FURS JSON Schema lists P7 as a required KPR field, while the FURS field-definition spreadsheet marks it optional. v1 follows the JSON Schema and treats it as required, since that is what the file must satisfy to validate.

Core-amount handling: Amount fields are technically optional in the schema, but a ledger line with no monetary values is not meaningful. v1 exposes only the common domestic-supply amount fields listed above (net base + VAT/deductible VAT by the standard rates). All other amount fields in the schema — intra-EU acquisitions/supplies, reverse-charge/self-assessment, distance sales, imports, exempt supplies, real-estate and fixed-asset breakdowns, non-deductible VAT, etc. — are out of scope for v1. An amount field left blank by the user is omitted from the JSON entry rather than written as 0.

7. Validation rules

Derived from the FURS JSON Schema (DDV_KIR_KPR_1.json) and the FURS validation-rules sheet (KIR_KPR_pravila.xlsx). v1 performs client-side format validation only; it does not perform any check that requires FURS online services (e.g. tax-register existence of a VAT ID).

RuleDetail
Tax number (TaxPayerID)Exactly 8 digits ^[0-9]{8}$.

OBDOBJE (MMMM)4 digits; start month ≤ end month; monthly or quarterly only (e.g. 0101, 0606, 0103, 0406, 0709, 1012).

Behavior on validation failure: the Download action is blocked (or the offending field is flagged) until all format rules pass, so the app never produces a structurally invalid file.

8. Download behavior

  • A single Download button assembles the current form state into one DDV_KIR_KPR JSON document and offers it to the user as a file download.
  • The button runs the §7 validations first; if any fail, download is prevented and the user is shown what to fix.
  • Blank optional amount fields are omitted from entries; the KIR/KPR header flags reflect which lists actually contain entries.
  • Amounts are written as JSON numbers with at most two decimal places.
  • Everything happens client-side; no data leaves the browser.

9. Non-goals (explicitly deferred)

The following are out of scope for the initial implementation:

  • DDV-O return generation, and any XML or CSV output — JSON KIR/KPR only.
  • Direct submission to FURS/eDavki — no REST/SOAP web-service call, no ZIP upload; the user submits the downloaded file themselves.
  • Self-disclosure (samoprijava) and other non-standard treatments — OBRAVNAVA is fixed to "1"; OBDOBJE88 and DAVEK88 are not produced.
  • Foreign taxpayer / tax-representative filing — TUJEC1, TUJEC2 and related refund/deductible restrictions are not produced.
  • Non-core amount fields — intra-EU, reverse-charge/self-assessment, distance sales, imports, exempt supplies, real-estate and fixed-asset breakdowns, non-deductible VAT, etc.
  • Tax-register (RDZ) lookups and any other validation requiring FURS online services.
  • User accounts, authentication, roles, authorization (e.g. filing on behalf of another taxpayer).
  • Persistence between sessions (no saved drafts, no database, no server-side storage).
  • Header extras: NACIN, INSPOS, PREDLODO, OPOMBA.

10. Legal / regulatory references and sources

  • ZDDV-1 (Zakon o davku na dodano vrednost) as amended by ZDDV-1O, introducing the obligation (from 1 July 2025) to submit the evidence of charged VAT and evidence of deducted VAT for each tax period.
  • Standard Slovenian VAT rates reflected in the fields: 22 % (standard), 9.5 % (reduced), 5 % (second reduced), 8 % (flat-rate compensation for farmers).
  • FURS technical package DDV_KIR_KPR.zip (JSON Schema DDV_KIR_KPR_1.json, example JSON, field mapping DDV_KIR_KPR_Polja.xlsx, validation rules KIR_KPR_pravila.xlsx), from the eDavki "KIR/KPR web service" area — the authoritative source for all field codes, mandatory-field sets, and formats in this spec (including VRACILO being an optional boolean with the single foreign-taxpayer dependency documented in §7).

11. Open questions

None blocking. Items deliberately deferred to a later version are captured in §9 (Non-goals). If a future version adds intra-EU/reverse-charge scenarios, the additional amount fields already exist in the same FURS schema and can be layered onto this structure without changing the file's overall shape. Note that if foreign-taxpayer filing (TUJEC1/TUJEC2) is ever brought into scope, the VRACILO refund flag must then be constrained to false for those filings (§7).
