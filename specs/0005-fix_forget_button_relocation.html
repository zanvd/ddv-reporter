Version: 1.0 (draft for implementation planning)

Date: 2026-07-17

Status: Confirmed — ready for implementation planning

Type: Bug fix — UI / layout only. No new feature, no tax logic.

Scope: Product / requirements only. No technical, architectural, or implementation decisions.

Builds on: Spec 0002 (Persist General (Glava) Data) — this refines the save/forget feedback behavior defined there. All of spec 0002's save/forget, storage, and validation behavior is otherwise unchanged.

Not a VAT rule. This is a UI layout fix in the Glava persist-controls row. It introduces no tax logic and changes nothing about how DDV is reported, calculated, or validated against FURS rules. There is no FURS / ZDDV-1 dependency to research or cite for the behavior described here (see §9).

1. Summary

Fix a layout bug in the Glava persist-controls row (.persist-row): the "Pozabi" (forget) button jumps horizontally whenever the save status message appears or disappears. Fix by placing the two buttons adjacent to each other, followed by a single shared feedback area to the right of both buttons that shows only the most recent action's message (save or forget) for 3 seconds. The forget button's on-screen position becomes stable and no longer depends on transient status-message visibility.

2. Background / the bug

Spec 0002 introduced the "Zapomni si" (save) / "Pozabi" (forget) action row with a transient feedback indicator next to each button. In the current layout the row renders four elements in this flex DOM order: 

[ "Zapomni si" button, save feedback, "Pozabi" button, forget feedback ]

Each feedback slot is a role="status" span that is hidden by default (display: none) and shown for 3 seconds after its button's action, then hidden again. Because the save feedback sits between the two buttons in the flex row, its appearance takes horizontal space and its disappearance releases it — which pushes the "Pozabi" button left/right depending on whether a save status message is currently showing.

That horizontal jump is the bug: the forget button's position depends on transient status-message visibility.

3. Scope

In scope

  • Reorder the persist-controls row so both buttons sit adjacent to each other first, with a single shared feedback area after both buttons, not interleaved between them (§4, R2).
  • Stable "Pozabi" position: the forget button must never move because a status message appeared or disappeared (§4, R1).
  • Last-action-wins feedback: only one message is visible at a time, showing the most recent action's outcome for 3 seconds (§4, R3).

Out of scope

  • Any change to the save / forget logic, local-storage persistence, or auto-restore behavior (all per spec 0002).
  • Any change to validation, the download gate, the DDV_KIR_KPR output, or FURS rules.
  • Any change to the message copy ("Shranjeno", "Pozabljeno", "Poskusite znova."), the button labels ("Zapomni si", "Pozabi"), the success/error icon styling, or the 3-second duration.
  • No new user-facing copy is introduced.

4. Requirements

R1 — Stable "Pozabi" position

The "Pozabi" button's on-screen position must be stable: it must never move because a status message appeared or disappeared. Since the shared feedback area sits after both buttons (R2), toggling its visibility can no longer shift either button — the reorder alone satisfies this requirement. No fixed-width reservation of the feedback area is required.

R2 — Single shared feedback area after both buttons

Reorder the row so the two buttons are adjacent, followed by one shared feedback area: 

[ "Zapomni si" button, "Pozabi" button, shared feedback area ]

Both the "Zapomni si" and "Pozabi" outcomes render into this same feedback area. The two separate, interleaved per-button feedback slots are replaced by this single shared area.

R3 — Last-action-wins, 3 seconds

  • At most one message is visible at a time.
  • A message stays visible for 3 seconds (the existing FEEDBACK_DURATION_MS = 3000 ms) from its action, then hides.
  • If either button (save or forget) is clicked again before the timer elapses, the new action's message immediately replaces the current one and the 3-second timer restarts.
  • Only the last action's message is ever shown — for 3 seconds from that action.

5. Preserved behavior (must not change)

Everything below is existing behavior from spec 0002 and must be preserved verbatim by this fix:

  • Success vs. error styling: green checkmark for a successful action; red circle-with-X for a failure — the existing visual language (success/error icon + text color) is unchanged.
  • Message copy, unchanged (no new copy): "Shranjeno" (successful save), "Pozabljeno" (successful forget), "Poskusite znova." (failed save or forget). All user-facing text remains Slovenian per project convention.
  • Accessibility: the shared feedback area keeps the role="status" live-region behavior so the latest message is announced.
  • "Pozabi" disabled state is still driven by hasSavedGeneral() and refreshed only after a successful save (→ enabled) or forget (→ disabled); a failed action leaves it unchanged.
  • The 3000 ms duration constant is unchanged.

8. Edge cases

CaseExpected handling
Rapid alternating save/forget clicks Only the latest outcome is shown in the shared area; the 3-second timer always reflects the most recent click (R3).
Same button clicked repeatedly before the timer elapses The message updates to the latest outcome and the timer restarts (R3).
A failed action after a prior success (or vice versa) within 3 s The shared area switches to the newest action's message (success ↔ failure) and restarts the timer. The "Pozabi" disabled-state refresh still occurs only on success (§5).
Message currently visible when its 3 s elapse with no new action The shared area hides; "Pozabi" position is unaffected (R1) since the area sits after both buttons.
