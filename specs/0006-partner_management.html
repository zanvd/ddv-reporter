DDV Reporter — Partner Management (Partnerji)

Version: 1.5 (revised — selecting the placeholder or a stale/removed partner in a partner-backed row's dropdown now clears the row's hidden identity fields to empty, instead of leaving them at whatever they last held)

Date: 2026-07-19

Status: Approved — ready for implementation planning

Type: New feature — data-management / UX convenience layered on the existing KIR/KPR entry.

Scope: Product / requirements only. No technical, architectural, or implementation decisions.

Builds on: Spec 0001 (KIR/KPR JSON export — the invoice fields and their format validation), Spec 0002 (local-only persistence model), Spec 0003 (tabbed interface & Pomoč help page), and Spec 0005 (last-action-wins transient feedback). All of those remain otherwise unchanged.

Not a new VAT rule. Partner management is a saved-address-book convenience for reusing customer/supplier identity data across invoices. It introduces no new tax logic, no new FURS fields, and no change to how DDV is calculated, serialized, or validated. The partner fields reuse the existing FURS-grounded validation from Spec 0001 (country code, VAT identification number). The DDV_KIR_KPR output format is unchanged: a partner merely supplies values into already-specified fields (P6/P6DS for KIR, P6/P7/P7DS for KPR). Everything stays in the browser — partner data never leaves the device (Spec 0002 privacy property).

1. Summary

Add a new "Partnerji" tab (positioned second, between "Poročilo" and "Pomoč") that lets the user maintain a saved list of business partners — each with an id, naziv (name), koda države (country code), and identifikacijska številka za DDV (VAT id). Partners are created, edited, and deleted inline in a per-row list, persisted locally in the browser, and surfaced via transient bottom-right message bubbles. On the "Poročilo" tab, the KIR and KPR lists each gain a second add button ("Dodaj izdani/prejeti račun partnerja") that inserts an invoice row whose customer/supplier identity fields are replaced by a single partner-selection dropdown, so the user reuses a saved partner instead of re-typing its data. The Pomoč tab gains a "Partnerji" help article, and the report-form instructions gain a short blurb about the two new partner buttons.

2. Background / motivation

A VAT payer typically issues to, and receives from, the same recurring customers and suppliers. Re-entering each partner's country code and VAT identification number on every invoice row is repetitive and error-prone. A saved partner list lets the user enter each partner's identity once and then pick it from a dropdown when adding invoices. Because the partner's country code and VAT id map directly onto existing FURS fields (§7), this is purely a data-reuse convenience — it changes nothing about what the DDV_KIR_KPR file contains or how it is validated.

3. Scope

In scope

  • A new "Partnerji" tab, second in the tab bar (order: Poročilo, Partnerji, Pomoč) (§4).
  • A partner list with inline add / edit / delete, an empty-state message, and an "add" (+) affordance (§5).
  • Local persistence of partners (create/update/delete), consistent with the app's existing local-only browser storage and privacy property (Spec 0002) (§5.6).
  • Transient bottom-right message bubbles (success/error, single slot, 3 s auto-dismiss) for the three partner actions (§6).
  • Field validation for partner name, country code, and VAT id, reusing the app's existing rules and inline-error UX (§8).
  • KIR/KPR integration: a second add button per list that inserts a partner-backed invoice row using a partner-selection dropdown in place of the identity fields (§9), including the decided selection behavior (immediate copy of partner data into the row; per-row dropdown snapshot frozen at creation; a persistent inline "Partner ne obstaja." indicator for a stale selection; an "Izberite partnerja." message for a missing selection; partner-backed rows validated by the same optional/required rules as manually-typed rows) (§9.3, §9.4).
  • Help content: a "Partnerji" article on the Pomoč tab, plus a blurb in the "Poročilo" instructions describing the two new buttons (§10).

Out of scope

  • Pagination, filtering, sorting controls, or search over partners (list is always the full set, ordered by id ascending).
  • Any change to the DDV_KIR_KPR output format, FURS fields, serialization, or the invoice validation rules themselves (Spec 0001 unchanged).
  • Deduplication / uniqueness enforcement across partners (two partners with the same VAT id, or the same name, are allowed) — not required in this version (§11).
  • Import/export of the partner list, sync across browsers/devices, or any server interaction (nothing leaves the browser).

4. Tab placement & navigation

  • The tab bar gains a third tab. Order becomes: Poročilo (first), Partnerji (second), Pomoč (third).
  • Tab label (Slovenian, fixed): "Partnerji".
  • Default tab on load stays "Poročilo". No deep-linking / URL persistence of the selected tab (consistent with Spec 0003).
  • Tab switching is state-preserving (show/hide, never a reset): switching away from and back to "Partnerji" preserves the list, any in-progress new/edited row, and its validation state — the same guarantee Spec 0003 gives "Poročilo". Likewise, switching to "Partnerji" and back must not disturb the "Poročilo" form.
  • Tabs remain keyboard-operable and expose the selected state to assistive technology, exactly as established in Spec 0003 (the existing tab mechanism simply gains one more tab).

5. Partner list (Partnerji tab)

5.1 Layout

  • The panel opens with an in-panel section heading, the fixed <h2>Partnerji</h2>, matching the per-section <h2> landmark pattern established in Spec 0003 (the Poročilo panel carries "Glava", "Izdani računi (KIR)", and "Prejeti računi (KPR)"; the Pomoč panel carries its help-article headings such as "Pomnenje"). The heading text is the fixed Slovenian string "Partnerji" — the same word as the tab label (§4).
  • Below the heading, the tab shows a list of saved partners, ordered by id ascending.
  • Each partner is a row with these columns, left to right. The four column headers are the fixed literal strings "ID", "Naziv", "Koda države", "ID za DDV", then two unlabeled (nameless) columns for the row-scoped save button and the row-scoped delete button. "ID za DDV" is the visible (abbreviated) header for the VAT-identification-number column; its full accessible label remains "Identifikacijska številka za DDV (brez kode države)", kept available for the field's aria-label / tooltip and for validation-rule naming (§7). No other column header is abbreviated.
  • All column headers / field labels are Slovenian (§7).
  • At the top-right of the list sits a "+" icon button to add a new partner (§5.5). Its accessible name is the fixed aria-label="Dodaj partnerja". Exact styling is left to implementation, but it must be a clearly actionable icon button.

5.2 Empty state

  • When no partners exist, the list area shows the message: "Brez shranjenih partnerjev."
  • The "+" add button remains present and usable in the empty state.

5.3 Fields & editability

  • A partner record consists of exactly four fields: id, naziv, koda države, identifikacijska številka za DDV.
  • id is never editable and is never entered by the user — it is assigned by the app (§5.5). It is displayed read-only in the ID column.
  • naziv, koda države, and identifikacijska številka za DDV are inline-editable form fields directly in the row:
    • Naziv — text input.
    • Koda države — the app's standard country dropdown (same control and option set as the KIR/KPR country field; emits ISO country codes, offers Greece once and emits GR).
    • Identifikacijska številka za DDV — text input.
  • All three editable fields are required, for both add and edit flows (§8).

5.5 Adding a partner (the "+" flow)

  • Clicking "+" scrolls the list into view (if not already visible) and appends a new empty row: all fields blank — including the normally read-only ID column, which is blank until the partner is saved and its id assigned.
  • The new row's two action buttons differ from a saved row: a green checkmark save button (icon-only, accessible name aria-label="Shrani partnerja") and, in place of the bin, a red "x" cancel button (icon-only, accessible name aria-label="Prekliči dodajanje partnerja").
  • Cancel on a new, unsaved row simply removes that row — no storage interaction, no message.
  • Save on a new row validates all three fields (§8). If valid: the partner is persisted with a newly assigned id, the id now appears in the row's ID column, the message "Partner dodan" is shown (§6), and the row's buttons revert to the normal saved state — the checkmark becomes the edit/update save button (now disabled until the next edit), and the cancel "x" becomes the normal red bin delete button. Their accessible names switch accordingly, from aria-label="Shrani partnerja" / aria-label="Prekliči dodajanje partnerja" to aria-label="Posodobi partnerja" / aria-label="Odstrani partnerja" (§5.4). If invalid, the row's field errors are revealed and nothing is persisted.
  • Save-button enabled state for a new row: unlike an existing row (whose save is gated on having an unsaved edit), a new row's save button is enabled immediately. Clicking it on an all-blank row therefore reveals the required-field errors, mirroring the report form's reveal-on-submit behavior. (Decision — confirm; see §12.)
  • Only one unsaved new row at a time: while an unsaved new-partner row is present, clicking "+" again does not add another row — it just scrolls to the existing unsaved row.

5.6 Id assignment & persistence

  • Ids are positive integers starting at 1, assigned by the app, never entered or edited by the user.
  • On page load, the app determines the last-used id from the stored partners; each newly added partner gets last-used + 1. If no partners are stored, the first added partner gets id 1.
  • Partner create, update, and delete are each persisted to the app's local browser storage, consistent with the existing local-only persistence model (Spec 0002): the data stays in this browser on this device and never leaves it. (How storage is keyed/structured is an implementation decision, not specified here.)
  • Id reuse note: because the last-used id is recomputed from the stored partners at each load, deleting the highest-id partner and reloading could let that id be assigned again to a later partner. This is acceptable for this version (ids are internal identifiers, not report data). Flagged so the implementation plan is aware, not to be "fixed" here.

6. Messages (bottom-right bubble)

The three partner actions each produce a transient message. Unlike the existing inline form messages, these render as a bubble in the bottom-right of the content window, colored by class (success vs. error background).

  • Single shared slot — no stacking. At most one bubble is visible at a time. A new message immediately replaces any currently showing one (last-action-wins, consistent with Spec 0005's shared-feedback model). Rationale: these messages carry no critical information, so a newer one simply supersedes an older one rather than queueing or stacking to fill the screen.
  • Auto-dismiss after 3 seconds (reuse the existing 3000 ms feedback duration). Replacing a bubble restarts the timer.
  • Bubbles are announced to assistive technology (a live region), consistent with existing transient feedback.

No other column header is abbreviated.

8. Validation & inline-error UX

  • Partner fields reuse the app's existing validation rules (the single source of truth in validate.js), same as the KIR/KPR identity fields:
    • Naziv — required; non-empty; max 250 characters (the app's standard required-text rule).
    • Koda države — required; must be a recognized country code (EU member state, SI, or XI), per the app's country-code rule.
    • Identifikacijska številka za DDV — required; max 25 characters; entered without the leading country-code prefix (no "SI" prefix), per the app's VAT-id rule.
    All three are required for both add and edit.
  • Inline-error UX matches the report form (CLAUDE.md validation-UX convention): a field's error is quiet until that field is touched (input/change or blur), and the row's save-button click acts as the row's submit gate, revealing all remaining errors in that row at once (analogous to how the Download button reveals every error in the report form). Errors display in the same inline style as existing form fields.
  • A row with any field error cannot be saved (no persistence, no success bubble); the errors are revealed instead. This applies equally to a new row and an edited existing row.
Consistency note (country/VAT pairing). In the report form, KIR's customer country + VAT id are an optional pair (both or neither), while a partner always requires both. Because a partner-backed KIR row always carries both a country and a VAT id, it inherently satisfies the KIR pairing rule — the partner cannot produce a half-filled pair.

9. KIR/KPR report-form integration

9.1 New per-list "add partner invoice" buttons

  • The KIR list and the KPR list each gain a second add button, positioned to the right of the existing "new receipt" button in that list:
    ListExisting buttonNew partner button (Slovenian, fixed)
    KIR (izdani računi)Dodaj izdani računDodaj izdani račun partnerja
    KPR (prejeti računi)Dodaj prejeti računDodaj prejeti račun partnerja
  • Each new button is disabled when there are no saved partners, and enabled when at least one saved partner exists. Its enabled state reflects the current saved-partner set (so adding the first partner on the Partnerji tab enables these buttons when the user returns to Poročilo).

9.2 Partner-backed invoice row

  • Clicking a partner button adds a new KIR/KPR invoice row that differs from a normal row: the identity fields are hidden and replaced by a single partner-selection dropdown:
    • KIR: hide Koda države kupca (P6) and Identifikacijska številka kupca za DDV (P6DS); show the partner dropdown instead.
    • KPR: hide Firma / ime in sedež dobavitelja (P6), Koda države dobavitelja (P7), and Identifikacijska številka dobavitelja za DDV (P7DS); show the partner dropdown instead.
  • Selecting a partner immediately populates the row's hidden identity fields with a copy of that partner's current data; from that point the row holds its own independent copy, not a live reference to the partner record (§9.4).
  • All other invoice fields (dates, document number, net value, VAT amounts, KPR flat rate, etc.) behave exactly as in a normal row.
  • A partner-backed row and a normal row may coexist freely in the same list; the report output is a single combined KIR/KPR list as today.

9.3 Partner dropdown

  • The dropdown displays partner names as the option labels and uses partner ids as the underlying values.
  • It offers the same placeholder option as the app's country dropdown — the fixed string "— izberite —". There is no separate "a partner must be selected" required rule that changes what blocks download: a partner-backed row is gated entirely through its hidden identity fields, under the exact same optional/required rules as a manually-typed row (§9.4). A partner-backed KIR row with nothing selected has both customer country and VAT id blank — a valid optional pair, exactly like a typed KIR row left blank; a partner-backed KPR row with nothing selected has blank supplier fields — which already fails KPR's unconditional required-field rules, exactly like a typed KPR row left blank. No new gating rule either way.
  • Re-selecting the placeholder clears the row. If the user selects the placeholder ("— izberite —") on a partner-backed row whose hidden identity fields currently hold data (copied from an existing partner, or left over from a prior selection), those hidden fields are cleared to empty and any "Partner ne obstaja." indicator is cleared. The row then behaves exactly like a freshly added, never-touched partner-backed row (§9.4).
  • Missing partner selection → "Izberite partnerja." When a partner-backed row is blocked by the Download gate because no valid partner has been selected (a KPR row whose supplier fields are consequently blank), the error message displayed next to that row's dropdown is the fixed Slovenian string "Izberite partnerja." — shown in place of the hidden identity fields' own error text (§9.4). It follows the app's standard field-error convention (quiet until touched, revealed together by the Download gate). Exactly how and where this message is wired is an implementation detail. (For a KIR row, both-blank customer identity is a valid optional pair, so a missing selection does not block download and this message does not appear — consistent with a typed KIR row left blank.)
  • The option list is a point-in-time snapshot frozen when the row is created (via the "Dodaj … partnerja" button), built from the saved-partner set as it stands at that moment. It is never re-synced when partners are subsequently added, edited, or removed on the Partnerji tab. Consequences: an older row may keep offering (and displaying as its selected value) an option for a partner that has since been removed or renamed; and a partner added after a row was created does not appear in that row's dropdown. A newly added partner-backed row always gets a fresh snapshot of the current set (§9.4, §11).
  • Selecting a partner that no longer exists → persistent inline "Partner ne obstaja." Because the option list is frozen, it may still offer a partner that has since been removed. If the user selects such a stale option, a persistent inline error — "Partner ne obstaja." — is shown next to that row's dropdown and stays visible until the user selects a different, currently-existing partner. It does not auto-dismiss on a timer; this is a per-row inline indicator. It is a live UI-level check — "does this row's selected partner id still exist in the current partner list?" — computed independently of the Download-gate field-error logic; it is not a validate.js field error (no change to the entry validators' signatures), and it does not by itself gate the Download button (§9.4). Selecting a stale option clears the row's hidden identity fields to empty — exactly like re-selecting the placeholder, above — rather than repopulating them from the removed partner's last-known data or leaving them at whatever they previously held.
  • Label (Slovenian): "Partner".

9.4 Partner data → invoice fields (decided: immediate copy, validated as a normal row)

  • On selection, copy immediately. The moment an existing partner is chosen in a partner-backed row's dropdown, that partner's current country code and VAT id — and, for KPR, its name — are copied into the row's hidden identity fields (the FURS fields of §7). From that point the row holds its own independent copy of those values, not a live reference to the partner record. Selecting the placeholder or a stale/removed partner instead clears the row's hidden identity fields to empty — any previously copied data is discarded, not retained (§9.3).
  • Hidden fields are what get serialized and validated. Building the DDV_KIR_KPR JSON and the Download-gate validation both operate on the row's hidden identity fields, using the exact same optional/required rules as a manually-typed row — no partner-specific carve-out and no separate "a partner has been selected" gating rule. If those hidden fields fail (a partner-backed KPR row whose supplier fields are still blank because a valid partner was never selected), the Download gate blocks exactly as it would for a typed row; if they pass (a partner-backed KIR row left blank — a valid both-blank pair), it does not block.
  • The row shows a single dropdown-level message, never the hidden fields' own error text. A partner-backed row's hidden identity fields are not rendered, so their own validation error text is never surfaced. Instead, at most one message appears next to the dropdown: if a "Partner ne obstaja." indicator is showing (a stale option was selected, §9.3), that message remains; otherwise, when the row is blocked for want of a valid selection, the message is "Izberite partnerja." (§9.3). The exact wiring of these messages is an implementation detail.
  • The "Partner ne obstaja." indicator does not itself gate download. It is the live UI-level check of §9.3, computed independently of the Download gate; whether the row blocks download is decided purely by the (now-cleared) hidden fields' normal validation. Because selecting a stale option clears those fields (§9.3), a KIR row showing the indicator has a blank customer-identity pair — a valid optional pair (§9.4) — so the report downloads despite the indicator showing; a KPR row showing the indicator has blank required supplier fields, so the Download gate blocks it, with the visible message staying "Partner ne obstaja." (message precedence, above).
  • Independent of later partner deletion. Because the row carries its own copy, deleting the partner from the Partnerji tab afterward does not affect an already-populated, untouched row: the report downloads normally with no error, and the serialized DDV_KIR_KPR output comes from the row's copied data, not from a live partner lookup at download time.
  • Independent of later partner edits. By the same rule, editing a partner after a row has copied its data does not retroactively change that row (the row keeps its copied snapshot). Re-selecting the same still-existing partner re-copies its then-current data.
  • New rows reflect the current set. A partner-backed row added after a deletion builds its dropdown from the current (post-deletion) partner set, so a deleted partner is never offered as a choice in a row created afterward (§9.3, §11).

13. References / notes

Partner management is a data-management / UX convenience feature. It introduces no new tax semantics: the partner fields reuse the country-code and VAT-identification-number validation already established (and FURS-grounded) in Spec 0001, and a partner only supplies values into the existing DDV_KIR_KPR fields (P6/P6DS for KIR; P6/P7/P7DS for KPR). Accordingly there is no new external FURS / ZDDV-1 dependency to research for the behavior specified here — the relevant FURS field definitions and validation rules are those already cited in Spec 0001. The relevant references are internal:

  • Spec 0001 — VAT records KIR/KPR JSON export (specs/0001-vat_records_kir_kpr_json_export.html): defines the KIR/KPR fields, the P6/P6DS/P7/P7DS mapping, and the country-code / VAT-id validation rules the partner fields reuse.
  • Spec 0002 — Persist general (Glava) data (specs/0002-persist_general_data.html): defines the local-only browser-storage model and privacy property (nothing leaves the browser) that partner persistence follows, and the failure-return behavior of the storage layer.
  • Spec 0003 — Tabbed interface & Pomoč help page (specs/0003-tabbed_interface_help_page.html): defines the tab bar, state-preserving tab switching, the per-section <h2> heading pattern, and the Pomoč help-article structure that the new "Partnerji" tab, its in-panel heading, and its help article extend.
  • Spec 0005 — Last-action-wins transient feedback (specs/0005-fix_forget_button_relocation.html): the single-slot, last-action-wins, 3-second transient-message model reused by the bottom-right message bubbles (§6).
  • Project conventions (repo-root CLAUDE.md): all user-facing text is Slovenian using FURS "Naziv polja" terminology; validate.js is the single source of truth for field errors; validation is quiet-until-touched with a reveal-all gate; KISS/YAGNI (no pagination, search, uniqueness, or import/export in this version).
