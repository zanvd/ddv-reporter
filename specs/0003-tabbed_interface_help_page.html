DDV Reporter — Tabbed Interface & Help (Pomoč) Page Specification

Version: 1.1 (draft for implementation planning)

Date: 2026-07-16

Status: Confirmed — ready for implementation planning

Scope: Product / requirements only. No technical, architectural, or implementation decisions.

Builds on: Spec 0001 (VAT records KIR/KPR JSON export) and Spec 0002 (Persist general (Glava) data) — the persistence feature described in the "Pomnenje" help article is defined in spec 0002.

Not a VAT rule. This feature is navigation and static help/instructional content only. It introduces no new tax logic, no new FURS fields, no serialization or validation changes, and changes nothing about how DDV is reported, calculated, or validated. Consequently there is no FURS / ZDDV-1 dependency to research or cite for the behavior described here (see §9). All existing FURS field definitions, validation rules, and the DDV_KIR_KPR output from specs 0001–0002 remain unchanged.

1. Summary

Introduce a two-tab interface above the existing single-page content. The current application content moves into a tab called "Poročilo"; a new tab called "Pomoč" holds a plain-language help page. The page title (<h1>) stays above the tabs. The selected tab is visibly indicated. The "Pomoč" tab opens with a short, layman's-terms description of the app plus an open-source note linking to the repository, followed by short titled help articles — for now exactly one, "Pomnenje", explaining the Glava data-persistence feature (spec 0002) in plain terms. As part of this change, the existing .app-intro paragraph on the "Poročilo" tab is repurposed from a "what is this app" description into brief instructional guidance on how to fill out the form (the "what is this app" role now lives on "Pomoč"). Purely a UI/UX change; nothing leaves the browser.

2. Background / motivation

The app has grown from a single data-entry form into something with its own behaviors (e.g. local persistence, spec 0002) that benefit from explanation. Rather than crowding the working form with prose, a dedicated "Pomoč" tab lets users read plain-language help without leaving the app or losing in-progress work, and gives a stable home for future help articles. Tabs also keep the primary working surface ("Poročilo") uncluttered. Moving the "what is this app" explanation to "Pomoč" frees the "Poročilo" intro paragraph to do a more useful job right where the user works — telling them how to fill out the form. The help page additionally surfaces the app's defining privacy property — that entered data never leaves the browser — where a first-time user will actually read it.

3. Scope

In scope

  • A tab bar with exactly two tabs, labelled "Poročilo" and "Pomoč", rendered below the page <h1> (§4).
  • Moving all current page content except the <h1> — the intro paragraph, the three sections (Glava, KIR, KPR), and the download section — into the "Poročilo" tab (§4, §5). Structural position and order are preserved; the sections and download flow are unchanged.
  • Rewriting the .app-intro paragraph text on the "Poročilo" tab: its structural position stays (first item under the tab), but its text changes from describing what the app is to brief instructional guidance on how to fill out the form (§5.1, §6.1). New fixed Slovenian copy is specified in §6.1.
  • A visible selected-tab indication, distinguishable by more than color alone, and exposed to assistive technology (§4).
  • A "Pomoč" page containing: (a) a short layman's-terms description of the app, (b) an open-source note with a link to the repository, and (c) short titled help articles (§5, §6).
  • Exactly one help article, "Pomnenje", explaining the Glava persistence feature in plain terms (§6.3).
  • State-preserving tab switching: switching tabs must not lose entered data or reset validation state (§4).

Out of scope

  • Deep-linking / URL persistence of the selected tab. No URL hash or query param; the app always opens on "Poročilo" (§4). Bookmarking/sharing a specific tab is explicitly not supported.
  • Any additional help articles beyond "Pomnenje" (the structure supports more later, but none are specified now).
  • Any change to the data model, FURS fields, validation rules, serialization, or the DDV_KIR_KPR output (specs 0001–0002 unchanged).
  • Any change to the persistence behavior itself — "Pomnenje" only describes the spec-0002 feature; it does not alter it.
  • A search, table of contents, collapsible/accordion articles, or more than two tabs.

4. Layout & navigation

  • The page <h1> "DDV Poročilo" stays above the tab bar and is shown on both tabs (it is the app title, not tab content).
  • Directly below the <h1> sits a tab bar with two tabs in this order: "Poročilo" (first) and "Pomoč" (second).
  • Default tab on load is "Poročilo". Every page load / reload starts on "Poročilo" (no memory of the previously viewed tab — consistent with the no-deep-linking decision).
  • Exactly one tab's content is visible at a time. Selecting a tab shows its content and hides the other's.
  • Selected-tab indication (fixed requirement): the currently selected tab must be visibly distinct from the unselected tab, and the distinction must not rely on color alone (e.g. also weight, an underline/fill, or similar non-color cue), so it is perceivable regardless of color vision. The selected state must also be conveyed to assistive technology (a screen-reader user must be able to tell which tab is active and which are available). Exact visual styling is left to implementation planning; it must reuse the app's existing visual language and palette (--accent, --border, --bg-soft) and introduce no bespoke component styling beyond what the tab bar requires.
  • State-preserving switching (fixed requirement): switching from "Poročilo" to "Pomoč" and back must preserve everything in "Poročilo" — entered Glava/KIR/KPR values, added ledger rows, validation state (including whether errors are currently revealed), and the download message. Switching tabs is a show/hide of already-mounted content, never a reset, re-render-from-blank, or re-validation. (Rationale: the help tab is reference material a user may consult mid-entry; consulting it must be cost-free.)
  • Keyboard & focus (product-level requirement): tabs must be reachable and operable by keyboard, not mouse/pointer only. The precise key model is an implementation/accessibility detail, but keyboard operability itself is required.

5. Tab contents

5.1 "Poročilo" tab

Contains today's page content in the same order and structural position: the intro paragraph (.app-intro), the Glava section (general data + the "Zapomni si" / "Pozabi" persist controls), the Izdani računi (KIR) section, the Prejeti računi (KPR) section, and the download section (the "Prenesi JSON" button and its message). The sections, fields, validation, and download flow are unchanged and simply relocated under the tab.

One text change: the .app-intro paragraph's copy is rewritten (its structural position is unchanged). Previously it described what the app is; that role moves to the "Pomoč" description (§6.1). On "Poročilo" it becomes brief instructional guidance on how to fill out the form — see the fixed Slovenian copy in §6.1. No other wording, field, behavior, or validation in this content changes.

5.2 "Pomoč" tab

The "Pomoč" tab, in order:

  1. A short layman's-terms description of what the app is and does (§6.1).
  2. An open-source note with a link to the repository (§6.2).
  3. A series of short, titled help articles; for now exactly one — "Pomnenje" (§6.3). Each article has a heading and a short body. The layout must accommodate more articles later without redesign, but none beyond "Pomnenje" are specified here.

6.3 Help article "Pomnenje"

Article heading (Slovenian) — fixed: "Pomnenje"

Article body (Slovenian) — fixed copy:

V razdelku Glava lahko svoje splošne podatke (davčno številko, izbrano obdobje poročanja in obe potrditveni polji) shranite, da jih ob naslednjem obisku ni potrebno znova vnašati.

S klikom na gumb »Zapomni si« aplikacija shrani trenutne podatke razdelka Glava. Ob naslednjem odprtju aplikacije v istem brskalniku se ti podatki samodejno izpolnijo. Ob ponovnem kliku na gumb se shranjeni podatki posodobijo s trenutnimi. S klikom na »Pozabi« shranjeno kopijo izbrišete.

Aplikacija si zapomni le podatke razdelka Glava; vnesenih računov (KIR in KPR) ne shrani.

Vaši podatki ostanejo izključno v vašem brskalniku in nikoli ne zapustijo vaše naprave. Prav tako se ne prenašajo ali usklajujejo med različnimi brskalniki, napravami ali profili. Shranjena kopija je na voljo samo v istem brskalniku na isti napravi.

Shranjena kopija se lahko izgubi, če počistite podatke brskalnika, v zasebnem oz. anonimnem (incognito) načinu pa se shranjevanje morda ne bo obdržalo.

Content notes (why the copy says what it says): it names the real controls "Zapomni si" (save) and "Pozabi" (forget); states that restore is automatic on the next visit but only in the same browser (no cross-browser/device sync); scopes persistence to the Glava section only (invoices are never saved); stresses the privacy property that data never leaves the browser; and closes with the two edge-case caveats (cleared browser data and private/incognito mode) in a single sentence. The stale-value validation nuance from spec 0002 §6.4 is deliberately omitted as too detailed for a layman article.

7. Edge cases

CaseExpected handling
Narrow / mobile viewport — the two-tab bar in a small width The tab bar must remain usable and readable at small widths (e.g. wrap or shrink gracefully); both tabs stay reachable. Exact responsive treatment is left to implementation, but the tabs must not overflow off-screen or become unreachable.
Long help content taller than the viewport The "Pomoč" content scrolls normally within the page; the <h1> and tab bar behave as ordinary page content (no special sticky behavior required).
Validation errors present in "Poročilo" while the user is on "Pomoč" Switching to "Pomoč" hides but does not clear or resolve them; switching back shows the exact prior validation state (§4, state-preserving switching).
Reload while on "Pomoč" App reopens on "Poročilo" (no tab-state persistence, §4). Not a bug — the confirmed behavior.
Repository link unreachable / offline Out of the app's control; the link is a normal external link. No special handling required.

8. Non-goals (explicitly deferred)

  • Deep-linking or any URL/hash reflection of the selected tab.
  • Additional help articles beyond "Pomnenje".
  • A help search, table of contents, or accordion/collapse interaction.
  • More than two tabs.
  • Any change to persistence behavior, FURS fields, validation, serialization, or the DDV_KIR_KPR output.

9. References / notes

This feature is navigation and static help/instructional content only. It has no dependency on any FURS form, the ZDDV-1 / ZDDV-1O statute, the Pravilnik o izvajanju ZDDV-1, or any EU VAT rule, and it does not change the DDV_KIR_KPR file, its fields, or its validation. Accordingly, no external legal or regulatory sources apply to the behavior specified here, and none are cited — a deliberate note, not an omission. The only relevant references are internal:

  • Spec 0001 — VAT records KIR/KPR JSON export (specs/0001-vat_records_kir_kpr_json_export.html): defines the sections, fields, and validation model that the "Poročilo" tab now hosts unchanged.
  • Spec 0002 — Persist general (Glava) data (specs/0002-persist_general_data.html): defines the persistence feature — the "Zapomni si" / "Pozabi" controls, auto-restore, Glava-only scope, and local-only privacy property — that the "Pomnenje" help article (§6.3) describes in layman's terms.
  • Project conventions (repo-root CLAUDE.md): all user-facing text is Slovenian; nothing leaves the browser; KISS/YAGNI (no speculative structure — e.g. only the one "Pomnenje" article, though the layout must accommodate more later).
