DDV Reporter — Dark Mode Theme Toggle Implementation Plan

Plan version: 1.0

Date: 2026-07-17

Implements spec: specs/0004-dark_mode_theme_toggle.html (v1.0, confirmed)

Builds on plans: impls/0002-persist_general_data.html (storage model reused for the theme key) and impls/0003-tabbed_interface_help_page.html (header/tab-bar layout the toggle slots into).

Status: Confirmed — approved for implementation, including the palette in §5.

Scope: Technical/architecture only. The spec is the binding source of requirements; this plan describes how to build exactly what it specifies.

Presentation-only, no logic change. This is a theming change. It touches no pure logic module (derive / validate / serialize / state / countries), no data model, no FURS field, no validation rule, and produces the identical DDV_KIR_KPR output as specs 0001–0003. It reuses the existing storage.js seam for a new, dedicated theme key; nothing new leaves the browser.
Settled decisions (confirmed with the user):
  1. Accessibility state = label-only. The toggle conveys state through a state-reflecting Slovenian aria-label (§7 of the spec); no aria-pressed (avoids the ambiguous "pressed = which theme?" semantics of a two-state theme toggle; the spec explicitly permits a state-reflecting name).
  2. Instant swap, no color transition. Toggling changes tokens instantly — nothing to animate, so prefers-reduced-motion is satisfied by construction (KISS). No global transition on color is introduced.
  3. Palette is confirmed. The light values are the app's shipped colors; the dark values below are contrast-verified and have been reviewed and approved — consistent with spec §11.4, which left exact dark-theme values to implementation planning.

1. Overview

Add a light/dark theme controlled by a single data-theme attribute on the <html> element. The active theme is resolved and applied before first paint by a small classic inline <script> in the document head (the only viable no-flash mechanism without a build step, since ES modules are always deferred). All colors already come from CSS custom properties; the plan adds three missing tokens (--bg, --surface, --on-accent) to replace the few hardcoded whites, then adds a single :root[data-theme="dark"] block that overrides the token values. A header toggle button (inline sun/moon SVG) flips the attribute, persists the choice via a new themeStore.js over the existing storage.js, and updates its accessible label. Guiding principles: KISS/YAGNI — exactly two themes, one control, one attribute, one override block, no framework, no new dependency.

3. Mechanism

3.1 No flash of the wrong theme (spec §5)

ES modules are always deferred, so main.js cannot run before the first paint. The only build-step-free way to apply the theme pre-paint is a classic, synchronous inline <script> in <head>, placed before the stylesheet link so the attribute exists when CSS first applies. It reads storage directly, resolves the theme (stored choice → OS preference → light fallback, spec §5), and sets the attribute: 

<script>
(function () {
  try {
    var theme = null;
    var raw = localStorage.getItem('ddvReporter.theme');
    if (raw) {
      var env = JSON.parse(raw);
      var t = env && env.data && env.data.theme;
      if (t === 'light' || t === 'dark') theme = t;
    }
    if (!theme) {
      theme = (window.matchMedia
        && matchMedia('(prefers-color-scheme: dark)').matches) ? 'dark' : 'light';
    }
    document.documentElement.setAttribute('data-theme', theme);
  } catch (e) {
    document.documentElement.setAttribute('data-theme', 'light');
  }
})();
</script>
Deliberate, flagged exceptions. (a) This is the only inline JS in the app and the only JS outside an ES module — a conscious departure from the "logic lives in modules" convention, unavoidable because no-flash requires pre-paint synchronous execution that deferred modules cannot provide. (b) It duplicates two contract strings with themeStore.js: the storage key ddvReporter.theme and the envelope path .data.theme. This minimal duplication is intentional; both sides must honor the same key and the same { version, data } envelope shape that storage.js writes (spec 0002). The whole block is wrapped in try/catch, so any storage failure silently falls back to light — the storage-failure-tolerance requirement (spec §5, §9) holds for load as well as save.

3.2 Swapping the CSS custom properties (spec §6)

:root keeps the light palette as the default, so the app still renders correctly with no attribute and no JS. A single :root[data-theme="dark"] block overrides the token values. Native form controls (the checkboxes, the date input, scrollbars) are switched with the CSS color-scheme property so they render in a matching dark style rather than a bright default: 

:root {
  color-scheme: light;
  --bg: #ffffff;  --surface: #ffffff;  --bg-soft: #f6f8fa;
  --fg: #1a1a1a;  --muted: #5a5a5a;    --accent: #0b5c8a;
  --border: #d5d9dd;  --error: #b3261e;  --success: #2e7d32;
  --on-accent: #ffffff;
}
:root[data-theme="dark"] {
  color-scheme: dark;
  --bg: #14181d;  --surface: #232b33;  --bg-soft: #1b2127;
  --fg: #e7eaed;  --muted: #a3adb6;    --accent: #6bb8e8;
  --border: #414b54;  --error: #f0a9a4;  --success: #7fd68a;
  --on-accent: #10161c;
}
html { background: var(--bg); }   /* fills the margins outside the centered body */

The toggle icon is chosen purely in CSS from the attribute — correct at first paint, no JS needed to pick it, so no icon flash: 

.theme-toggle .icon-moon { display: none; }              /* default (light): show sun */
:root[data-theme="dark"] .theme-toggle .icon-sun  { display: none; }
:root[data-theme="dark"] .theme-toggle .icon-moon { display: block; }

3.3 The toggle control — placement, markup, wiring (spec §4, §7, §8)

The existing <h1> and the new button share a header line in a flex wrapper, a sibling above the tab bar — outside both tab panels, so it is visible on every tab (spec §4). The button is a native <button> (keyboard-operable for free), with a decorative icon hidden from assistive tech and a Slovenian action-worded accessible name: 

<div class="app-header">
  <h1>DDV Poročilo</h1>
  <button type="button" id="theme-toggle" class="theme-toggle"
          aria-label="Preklopi na temni način">
    <span class="icon-sun"  aria-hidden="true"><svg …></svg></span>
    <span class="icon-moon" aria-hidden="true"><svg …></svg></span>
  </button>
</div>

<div class="tabs" role="tablist" …> … </div>   ← unchanged, still below the header
  • Icon reflects the current theme (sun while light, moon while dark) via the CSS in §3.2. Both SVGs are inline, self-contained (no external asset / icon font), consistent with the no-dependency architecture (spec §4).
  • Accessible name reflects the destination action: light active → "Preklopi na temni način"; dark active → "Preklopi na svetli način" (spec §7). The static HTML hardcodes the light-mode label; themeToggle.js corrects it on init from the already-applied attribute (an aria-label is not painted, so there is no visible flash, and screen readers read it on focus — after init has run).
  • The SVG glyphs carry aria-hidden="true" so only the button's name is announced (spec §7, §8).
  • Focus indicator: .theme-toggle:focus-visible uses the accent-colored outline, matching the tab pattern; it meets the §6 contrast target in both themes (see §5 table).

The controller is tiny and DOM-only:

// view/themeToggle.js
import { saveTheme } from '../themeStore.js';

const LABEL = {
  light: 'Preklopi na temni način',   // light active → offer dark
  dark:  'Preklopi na svetli način',  // dark active  → offer light
};

export function initThemeToggle(buttonEl) {
  const root = document.documentElement;
  const sync = () => buttonEl.setAttribute('aria-label', LABEL[root.dataset.theme] || LABEL.light);
  sync();                                   // correct the static label to the resolved theme
  buttonEl.addEventListener('click', () => {
    const next = root.dataset.theme === 'dark' ? 'light' : 'dark';
    root.dataset.theme = next;              // instant visual swap (all tokens re-resolve)
    saveTheme(next);                        // failure-tolerant; return value ignored (spec §5)
    sync();
  });
}

The handler changes only the attribute, storage, and the label. It does not read or write app data, validation state, the active tab, or scroll position, and moves no focus programmatically — so toggling mid-entry changes colors only (spec §5, §9). Because only color tokens change (no box sizes), there is no reflow.

3.4 Persistence — themeStore.js (spec §5, reusing spec 0002)

// themeStore.js
import { readRecord, writeRecord } from './storage.js';

const DOMAIN = 'theme';   // → localStorage key 'ddvReporter.theme'
const VERSION = 1;
const VALID = ['light', 'dark'];

export function loadTheme() {
  const record = readRecord(DOMAIN);
  const theme = record && record.data && record.data.theme;
  return VALID.includes(theme) ? theme : null;
}

export function saveTheme(theme) {
  return writeRecord(DOMAIN, VERSION, { theme });
}
  • Its own dedicated key under the ddvReporter.* namespace — independent of and not co-mingled with the Glava record (spec §5). No migration, no shared blob.
  • Reuses storage.js's try/catch-wrapped, failure-tolerant access verbatim: a failed saveTheme returns false, which the toggle ignores — the theme still changes for the session, it just may not be remembered (spec §5, §9). No user-facing error.
  • No forget/remove is provided — the spec has no clear-theme requirement (YAGNI). loadTheme() is used only by the inline head script's logic in spirit; the module itself is consumed by themeToggle.js (save) and by the unit test. The head script inlines the equivalent read because it cannot import a module (§3.1).

3.5 Data flow

page load
  └─ <head> inline script (sync, pre-paint)
        reads localStorage 'ddvReporter.theme' → else prefers-color-scheme → else light
        sets <html data-theme="…">                     ← CSS + correct icon apply on first paint (no flash)

  └─ stylesheet applies :root or :root[data-theme="dark"] token set

  └─ main.js (deferred module) mounts views, then:
        initThemeToggle(#theme-toggle) → syncs aria-label to the resolved theme, wires click

user clicks toggle
  └─ flip <html data-theme> (instant swap) → saveTheme() (best-effort) → re-sync aria-label

4.4 New rules

  • html { background: var(--bg); } and body keeps color: var(--fg).
  • .app-header { display:flex; align-items:flex-start; justify-content:space-between; gap:1rem; } — title left, toggle at the right edge of the content column; wraps safely on narrow widths so the toggle never overflows (spec §4 responsiveness).
  • .theme-toggle: transparent background, 1px solid var(--border), rounded, square-ish tap target (~2.25rem), color: var(--fg) (the SVGs use fill/stroke: currentColor so they recolor with the theme); .theme-toggle:focus-visible { outline:2px solid var(--accent); outline-offset:2px; }.
  • The icon show/hide rules from §3.2.
  • ::placeholder { color: var(--muted); } so any input placeholder stays legible in dark mode (spec §6 "placeholders remain legible").
No color transition is added (settled decision 2). The swap is instant, so there is nothing for prefers-reduced-motion to gate — the requirement is met by construction. If a fade is ever wanted later, it must be wrapped in the existing @media (prefers-reduced-motion: reduce) guard.

5.3 Verified WCAG 2.1 AA contrast

Ratios below are computed from WCAG relative luminance (sRGB linearization), not eyeballed. Targets: normal text ≥ 4.5:1, large text / meaningful UI ≥ 3:1.

PairLightDarkTarget
--fg on --bg17.4:114.8:14.5
--fg on --surface17.4:111.9:14.5
--muted on --bg6.9:17.8:14.5
--muted on --surface6.9:16.3:14.5
--accent text on --bg7.2:18.2:14.5
--accent on --surface7.2:16.6:14.5
--on-accent on --accent (download button)7.2:18.4:14.5
--error text on --bg6.5:19.3:14.5
--error on --surface6.5:17.5:14.5
--on-accent on --error fill (badge glyph)6.5:19.5:14.5
--success on --bg5.1:110.1:14.5
--accent focus outline on --surface7.2:16.6:13.0
Resting input borders are subtle by design in both themes. The shipped light theme's --border on white is ~1.4:1; the dark --border on --surface is ~1.6:1 (i.e. at least as distinguishable as the accepted light baseline). The AA-critical 3:1 affordance for form fields is carried by the focus outline (6.6:1) plus the --surface-vs---bg-soft fill difference and the always-present field label — not by the resting border. This matches how the already-accepted light theme behaves; it is called out here so the trade-off is explicit rather than hidden. If reviewers want stronger resting field borders in dark mode, lightening --border toward ~#5b656f reaches ~3:1 on --surface without affecting anything else — a palette-only tweak.

7. Testing

  • Unit — themeStore.test.js (Node --test, mirrors generalStore.test.js, injecting a fake backend / stubbing globalThis.localStorage):
    1. saveTheme('dark') then loadTheme() → 'dark' (round-trip through the { version, data } envelope).
    2. loadTheme() with nothing stored → null.
    3. Stored but invalid value (e.g. 'blue', or a non-theme envelope) → null.
    4. Backend that throws on write → saveTheme returns false (failure-tolerance).
  • Existing unit suite stays green — no logic module is touched. Run cd src && npm test.
  • Smoke — extend .claude/skills/run/smoke.mjs to assert:
    1. On load, <html> has a data-theme of light or dark (default resolution ran), and no console errors.
    2. Clicking #theme-toggle flips data-theme, swaps the visible icon (sun↔moon), and updates aria-label to the matching Slovenian string.
    3. The toggle is keyboard-focusable and shows a focus outline.
    4. State preservation: type a Glava value, switch to Pomoč, toggle the theme, switch back — the value, the active tab, and any revealed error are unchanged (spec §5, §9).

8. Build sequence

  1. styles.css — add the three tokens + color-scheme, the :root[data-theme="dark"] block, html background, and swap the hardcoded #fffs (§4). App looks identical in light; dark is reachable by manually setting the attribute.
  2. index.html — add the pre-paint inline head script (§3.1) and the .app-header wrapper with the toggle button + inline SVGs (§3.3), plus the .theme-toggle/icon CSS.
  3. themeStore.js — the store module (§3.4).
  4. view/themeToggle.js — the controller (§3.3).
  5. main.js — import and call initThemeToggle(...).
  6. themeStore.test.js + smoke.mjs — add tests (§7); run unit + smoke.

9. Assumptions & out-of-scope

  • Out of scope (spec §3): more than two themes, user-customizable colors, per-tab/per-section theming, cross-device sync, live-following the OS after an explicit choice, a second entry point / settings panel, print optimization (spec §9/§11.4), and any change to data model, FURS fields, validation, serialization, or the DDV_KIR_KPR output.
  • Assumptions: (a) the theme key is ddvReporter.theme using the same { version, data } envelope storage.js already writes — the inline head script and themeStore.js both depend on this shared contract (§3.1 flag); (b) the two inline SVGs (sun, moon) are authored locally with currentColor fills so they recolor with the theme — no external icon asset is introduced (spec §4); (c) the dark hex values in §5 were left to implementation planning by spec §11.4 and have been reviewed and confirmed here.
