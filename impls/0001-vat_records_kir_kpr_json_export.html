DDV Reporter — Initial Implementation Plan

Plan version: 1.0

Date: 2026-07-15

Implements spec: specs/0001-vat_records_kir_kpr_json_export.html (v1.1, confirmed)

Status: Approved — ready to build

Scope: Technical/architecture only. The spec is the binding source of requirements; this plan describes how to build exactly what it specifies.

Settled decisions (approved by user):
  1. Vanilla HTML/CSS/JS, no build tooling, no framework, no runtime dependencies.
  2. Hand-rolled format validation (no ajv, no bundled FURS JSON Schema).
  3. Header flags KIR, KPR, VRACILO, ODBDELEZ serialize as native JSON true/false.

1. Overview

A fully client-side, single-page web application. The user fills a form using human-readable Slovenian labels; the application keeps all state in memory, runs client-side format validation, and on Download assembles a single DDV_KIR_KPR JSON document (using FURS field codes) and hands it to the browser as a file. There is no backend, no persistence, and no network traffic — the spec's non-goals (§9) make this a pure front-end artifact that can run by opening a local file. Guiding principles: KISS / YAGNI — expose only the minimal field subset from spec §5–§6, hardcode what the spec fixes (OBRAVNAVA = "1"), and auto-derive everything the spec marks as derived (ZAPST, OBDOBJE, OBDOBJE_OD/OBDOBJE_DO, and the KIR/KPR flags).

2. Technology

ChoiceJustification (traced to spec)
Vanilla HTML5 + CSS + ES2020 JavaScript (ES modules), no build step Spec §1/§8 require a standalone, client-side app where "no data leaves the browser." Zero-build vanilla is the simplest thing that satisfies this; it runs from the local filesystem and mirrors how the spec itself ships (a self-contained HTML file). KISS/YAGNI: the minimal v1 field set does not warrant a framework or bundler.
No runtime dependencies All logic (derivation, validation, serialization, download) is expressible with the standard DOM + Blob/URL APIs. Nothing in the spec requires third-party code.
Hand-rolled validation The §7 rules are a small, closed set of regex / numeric-range / date / enum / pairing checks over the v1 subset. A schema validator (ajv) plus the full FURS schema would pull in a build toolchain and validate far more than v1 produces — rejected on YAGNI grounds.
Static country-code table (inline JS constant) §7 fixes the allowed set: EU member states + SI + XI, with GR/EL both accepted for Greece. A hardcoded array is sufficient and needs no lookup service (§9 forbids FURS online lookups).
Tooling (dev only, optional): a static file server for local testing; a lightweight test runner for the pure modules Not shipped with the app. Only used to exercise the pure logic modules (§9 of this plan). Does not add a runtime dependency or a build step to the deliverable.
Browser APIs relied upon: DOM, <input type="date"> and <select> for structured entry, Blob + URL.createObjectURL + a programmatic <a download> click for the file save, and JSON.stringify for serialization. All are broadly available in current evergreen browsers.

3. Domain model

Three plain in-memory objects: one header and two arrays of entries. Form state stores raw user input plus a period selection; FURS-coded values are computed only at serialization time. The two entry types are modeled separately — the spec reuses field codes with different meanings across the lists (e.g. KIR P7 = net amount vs. KPR P7 = supplier country code), so conflating them would be a source of bugs.

Fields marked ? are omitted entirely when the user leaves them blank (§6, §8) — never emitted as 0 or "". When a list has no entries, its header flag is false and the empty list is carried as an empty array (§4).

4. Architecture

Small module set, each with a single responsibility. Logic that must be correct (derivation, validation, serialization) is isolated into pure functions with no DOM access, so it is directly unit-testable. The view layer is the only part that touches the DOM.

Data flow: user input → state → live validate (inline feedback) → on Download: validate everything (gate) → if clean: derive → serialize → download. Validation lives in exactly one module and is invoked both live and as the Download gate, so the app can never emit a format-invalid file (§7, §8).

5. Key implementation concerns

5.1 Period derivation

  • Monthly: OBDOBJE_OD = first day of month; OBDOBJE_DO = last day of month (calendar-correct, incl. leap Feb); MMMM = MM+MM (e.g. June → 0606).
  • Quarterly: quarter n → start month 3n−2, end month 3n; OBDOBJE_OD/OBDOBJE_DO span the quarter; MMMM = start+end month (e.g. Q2 → 0406).
  • The single derived OBDOBJE is written identically onto every KIR and KPR entry (§6).

5.2 Validation strategy (§7)

RuleImplementation
TaxPayerIDRegex ^[0-9]{8}$.
AmountsParse as number; reject >2 decimal places; enforce strictly within (−5e9, +5e9); allow negatives. Store/emit as JSON number.
Dates (P2,P4,P5)YYYY-MM-DD shape + real calendar date; line dates not in the future (compared to local today).
OBDOBJE/period boundsDerived, so structurally always valid; a guard asserts each entry's derived OBDOBJE equals the header's.
ZAPSTGuaranteed consecutive by renumbering on every add/remove; not user-editable.
Country codesMembership test against the static allowed set; uppercase; accept GR and EL for Greece.
KIR P6/P6DS pairingBoth present or both empty; otherwise a paired-field error.
VAT-ID format≤25 chars; reject a leading country-code prefix pattern to nudge correct entry.
Text fields≤250 chars.
Required fieldsPresence enforced per §5/§6 before Download is allowed.
VRACILO foreign-taxpayer ruleDocumented as inactive in v1 (no TUJEC* produced); not implemented (§7, §9).

On failure, the Download action is blocked and offending fields are flagged inline with a message (§7, §8). Validation is presentation-agnostic (returns data, not DOM), keeping it testable.

5.3 Serialization correctness

  • Amounts emitted as JSON numbers, ≤2 decimals; blank optional amounts omitted, not zeroed (§6, §8).
  • OBRAVNAVA always the string "1" (§6).
  • Header flags KIR/KPR/VRACILO/ODBDELEZ as JSON booleans (settled decision 3).
  • Empty list ⇒ empty array + flag false (§4).

5.4 Edge cases from the spec

  • Only-KIR or only-KPR filing: other list empty, its flag false (§4/§5).
  • Greece GR vs EL equivalence (§7).
  • Leap-year February end-of-month in quarterly/monthly period bounds.
  • Future-date rejection uses the browser's local date; documented as such.
  • Number formatting: guard against floating-point artifacts by validating decimal-place count on the raw input string before converting.

5.5 Testing approach

  • Unit tests target the pure modules — derive, validate, serialize, countries — which carry all the risk-bearing logic.
  • A golden-file test serializes a known fixture (mix of KIR + KPR, blank optionals, only-KIR case) and compares against a hand-verified expected JSON.
  • Optional dev-only cross-check: validate the golden output against the FURS DDV_KIR_KPR_1.json schema as an external oracle — not shipped, not a runtime dependency.

6.1 Rough work sequencing

  1. derive.js + countries.js (pure, no UI) with unit tests.
  2. validate.js (§7 rules) with unit tests.
  3. serialize.js with golden-file test.
  4. state.js (in-memory operations, ZAPST renumbering).
  5. view/* + index.html + styles.css — header, then the two lists, then inline errors.
  6. download.js and the Download gate wiring.
  7. End-to-end manual pass across the §5.4 edge cases.

7. Assumptions and out-of-scope

Assumptions

  • Target is current evergreen desktop browsers (the app may be opened from the local filesystem).
  • "Not in the future" is evaluated against the user's local system date.
  • Header booleans serialize as JSON true/false (settled decision 3); "(DA/NE)" in the spec is the human yes/no meaning, not a string literal in the JSON.
  • The generated download filename is a reasonable app-chosen default (e.g. incorporating tax number and period); the spec does not prescribe it.

Out of scope (restating spec §9 as it constrains the design)

  • DDV-O return; any XML or CSV output — JSON KIR/KPR only.
  • Direct submission to FURS/eDavki (no REST/SOAP, no ZIP upload).
  • Self-disclosure — OBRAVNAVA fixed to "1"; OBDOBJE88/DAVEK88 not produced.
  • Foreign-taxpayer / representative filing — TUJEC1/TUJEC2 and the dependent VRACILO rule not produced.
  • Non-core amount fields (intra-EU, reverse-charge, imports, exempt, real-estate/fixed-asset, non-deductible VAT, etc.).
  • Tax-register (RDZ) / online-service validation.
  • Accounts, authentication, authorization.
  • Persistence between sessions (no drafts, no storage, no server).
  • Header extras NACIN, INSPOS, PREDLODO, OPOMBA.
