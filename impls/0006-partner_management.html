DDV Reporter — Partner Management (Partnerji)

Type: Technical / architecture implementation plan (the how).

Date: 2026-07-19

Source spec: specs/0006-partner_management.html (v1.5, Approved).

Status: Approved for implementation — decisions 1–3 resolved as option A.

Builds on the existing codebase: Spec/impl 0001 (KIR/KPR fields, validators, serialize), 0002 (storage.js + generalStore.js local persistence), 0003 (tab mechanism, per-section <h2>, Pomoč articles), 0005 (last-action-wins 3 s feedback).

Guiding principle. Partner management is a data-reuse convenience, not a tax-logic change. The plan adds no new FURS field, and changes neither serialize.js nor the entry validators' output: a selected partner only ever reaches the wire by populating the row's existing identity keys (customerCountry/customerVatId for KIR; supplierName/supplierCountry/supplierVatId for KPR). Everything stays client-side; partner data is persisted with the same local-only storage.js envelope as Glava. KISS/YAGNI throughout: no dedup, pagination, search, sort, or import/export.

1. Overview

Add a saved partner address book layered over the existing KIR/KPR entry app. A new Partnerji tab (second) maintains a locally-persisted partner list with inline add/edit/delete and a bottom-right message bubble. On the Poročilo tab, each of the KIR and KPR lists gains a second "add" button that inserts a partner-backed invoice row whose identity fields are replaced by a single partner-selection dropdown. New logic lives in DOM-free modules (state.js, validate.js, a new partnerStore.js); new UI lives under view/. The generic entry-list renderer is extended with a config-gated partner branch; the Partnerji tab gets its own dedicated view. No change to serialize.js, derive.js, countries.js, storage.js, or tabs.js.

3. Domain model

3.1 Partner value object

Internal code names (English) map to the spec's fixed Slovenian labels and, when a partner is applied to an invoice, to the already-specified FURS fields (spec §7). None of the partner fields introduce a new FURS code.

Internal nameLabel — visible (Slovenian)TypeValidation (reused, §5)FURS when applied
idIDpositive integer, app-assignednone — not user-entered, not serialized— (app-internal)
nameNazivstringvalidateRequiredText (max 250)KPR P6; KIR: dropdown label only (not serialized)
countryCodeKoda državeISO code (emits GR for Greece)validateCountryCode({required:true})KIR P6 / KPR P7
vatIdID za DDV (accessible label: "Identifikacijska številka za DDV (brez kode države)")stringvalidateVatId({required:true})KIR P6DS / KPR P7DS

Field names are settled by the spec (fixed Slovenian labels) — no placeholder labels remain to confirm. The "ID za DDV" column header is the only abbreviated one; its full string stays available as the input's aria-label (§5.1, §7).

3.2 State shape

createState() gains two fields alongside the existing general/kir/kpr:

{
  general: { ... },        // unchanged
  kir: [ ... ],            // unchanged
  kpr: [ ... ],            // unchanged
  partners: [],            // NEW — [{ id:number, name, countryCode, vatId }], ordered by id asc
  nextPartnerId: 1,        // NEW — session id sequence (see §3.3)
}
Serialize/validate isolation. serialize.js and validate.js read only general/kir/kpr; state.partners and state.nextPartnerId are invisible to the DDV_KIR_KPR output and the Download gate. Partner-backed rows carry no extra keys in the kir/kpr entry objects — the frozen dropdown snapshot and "partner-backed" flag live in the view-layer row closure (§6.3), so the serialized output is byte-for-byte unchanged from a manually-typed row with the same identity values.

3.3 Id assignment (spec §5.6)

  • Positive integers starting at 1, never user-entered.
  • nextPartnerId is a session sequence seeded at load to max(stored ids) + 1 (or 1 if none). Each addPartner assigns the current value and increments it. This reproduces the spec's semantics exactly: no in-session id reuse (the sequence only climbs within a session), and reuse is possible only after a reload re-seeds the sequence from a now-lower stored max (e.g. after deleting the highest-id partner). Acceptable per §5.6 (ids are internal).

4. Technology

No change to the stack: vanilla HTML + CSS + ES-module JS, no build step, no framework, no runtime dependencies (CLAUDE.md). Persistence reuses the existing localStorage envelope in storage.js. Tests use Node's built-in runner for the pure logic modules; DOM-heavy view paths are exercised with the CDP smoke skill (.claude/skills/run/). No new tooling or libraries — every new concern is covered by existing primitives (touch.js, fieldBuilders.js builders, storage.js, the 3000 ms feedback pattern from persistControls.js).

5. Architecture — modules

Legend: new changed unchanged

5.1 js/partnerStore.js new

Partner-specific persistence glue, mirroring generalStore.js, on top of the generic storage.js. DOM-free.

  • Domain key 'partners' → localStorage key ddvReporter.partners; envelope { version: 1, data: { partners: [...] } }.
  • loadPartners() → sanitized array (each item reduced to the four recognized fields, unknown fields ignored — same defensive picking as generalStore.pickFields), or [] if nothing stored / record corrupt.
  • savePartners(partners) → boolean. Whole-list write-through: the entire array is re-persisted on every create/update/delete (simplest correct model; the list is small and never paginated). Returns storage.writeRecord's boolean so callers can drive the success/error bubble.
  • No validation gate here (validation happens in the view before calling save, §7).

5.2 js/state.js changed

  • createState() adds partners: [] and nextPartnerId: 1.
  • addPartner(state, { name, countryCode, vatId }) → assigns id = state.nextPartnerId++, pushes { id, name, countryCode, vatId }, returns the new id. (Partners are ordered by id ascending by construction; no renumbering, unlike KIR/KPR zapSt.)
  • updatePartner(state, id, patch) → merges editable fields into the matching partner.
  • removePartner(state, id) → splices the partner out. nextPartnerId is not decremented (§3.3).
  • seedPartners(state, partners) (or equivalent in main.js bootstrap) — replaces state.partners with the loaded array and sets state.nextPartnerId = max(ids)+1 || 1.

5.3 js/validate.js changed

  • Add validatePartner(partner) → field-error map keyed name/countryCode/vatId, reusing the existing validateRequiredText, validateCountryCode({required:true}), and validateVatId({required:true}). All three required for both add and edit (§8).
  • Add a tiny hasPartnerErrors(errorMap) helper (or reuse Object.keys(...).length > 0 at call sites).
  • No change to validateKirEntry/validateKprEntry signatures or output (spec §12.2). The "Partner ne obstaja." indicator is deliberately not a validate.js error — it is a live UI-level check in the view (§6.4).

5.4 js/view/partnerList.js new

The Partnerji tab view. Renders into #partner-content, bound to state.partners.

  • Layout (§5.1): a table with header row ID / Naziv / Koda države / ID za DDV + two headerless columns (save, delete). A top-right + icon button, aria-label="Dodaj partnerja".
  • Empty state (§5.2): "Brez shranjenih partnerjev." shown when state.partners.length === 0; the + button stays present and usable.
  • Per-row fields (§5.3): Naziv (text, maxlength 250), Koda države (the shared country select — reuse buildCountrySelect from fieldBuilders.js), ID za DDV (text, maxlength 25, with the full accessible aria-label). ID column read-only. Each editable field carries a .field-error element for inline errors.
  • Quiet-until-touched (§8): reuse view/touch.js per row; a field's error stays hidden until touched; the row's save-button click reveals all of that row's errors at once (the row-level analogue of the Download gate).
  • Saved-row buttons (§5.4): green checkmark save (aria-label="Posodobi partnerja"), disabled unless the row has an unsaved edit (at least one editable field differs from the persisted value — dirty-tracking against a per-row snapshot of the last-saved values); red bin delete (aria-label="Odstrani partnerja"), always enabled, no confirm dialog.
  • Add flow (§5.5): + scrolls the list into view and appends one blank row (ID blank) with a green checkmark save (aria-label="Shrani partnerja") and a red "x" cancel (aria-label="Prekliči dodajanje partnerja"). The new row's save is enabled immediately (spec §12.1 confirmed) — clicking it blank reveals required-field errors. Cancel just removes the row (no storage, no message). Only one unsaved new row at a time: a second + click scrolls to the existing unsaved row instead of adding another.
  • On successful save of a new row: assign id, show it in the ID column, and morph the two buttons to the saved-row pair (checkmark → disabled "Posodobi partnerja"; "x" → "Odstrani partnerja"), switching their aria-labels accordingly.
  • Persistence + feedback: each CRUD calls the state.js mutator then partnerStore.savePartners(state.partners). Write-through and rollback: apply the mutation to state.partners, attempt save; on false revert the mutation and show the action-specific error bubble; on true keep it and show the success bubble (§6, §11). After any successful CRUD, fire the injected onPartnersChanged callback (§5.7).
  • Validation gate: a row with any validatePartner error cannot be saved — errors are revealed inline, nothing is persisted, no bubble (a validation failure never reaches storage, distinct from a persistence failure, §6/§8).

Messages produced by this view (all fixed Slovenian, §6): success "Partner dodan" / "Partner posodobljen" / "Partner odstranjen"; persistence-failure "Pri dodajanju partnerja je prišlo do napake." / "Pri posodabljanju partnerja je prišlo do napake." / "Pri odstranjevanju partnerja je prišlo do napake."

5.5 js/view/messageBubble.js new

  • Single shared bottom-right bubble, mounted to a container in index.html. Exposes showMessage(text, { success }).
  • Last-action-wins single slot: a new message immediately replaces the visible one and restarts the timer; auto-dismiss after 3000 ms (reuse the existing duration). success/error class drives the background color.
  • Announced to assistive tech via a role="status" live region (consistent with persistControls feedback). Structurally close to createFeedbackSlot() in persistControls.js, but positioned as a global fixed bubble rather than an inline slot.

5.6 js/view/entryList.js + js/view/fieldBuilders.js changed

Partner-backed rows are added as a config-gated branch — inert unless config.partner is present (so Glava-less behavior for lists without partners is impossible; KIR/KPR both supply it).

config.partner descriptor (supplied by kirList.js/kprList.js)

partner: {
  addLabel: 'Dodaj izdani račun partnerja',        // or 'Dodaj prejeti račun partnerja'
  identityKeys: ['customerCountry', 'customerVatId'],   // KPR: ['supplierName','supplierCountry','supplierVatId']
  copyFrom: (p) => ({ customerCountry: p.countryCode, customerVatId: p.vatId }),
                                                   // KPR also copies supplierName: p.name
  currentPartners: () => state.partners,           // live set, for snapshot + stale check
}

Second add button (§9.1)

  • When config.partner exists, render a second button labeled config.partner.addLabel to the right of the existing add button. Disabled when currentPartners().length === 0. Its enabled state is refreshed by onPartnersChanged (§5.7).
  • Click: id = config.addEntry(state, emptyFields), then addRow(id, { partnerBacked: true, partnerOptions: snapshot() }) where snapshot() = currentPartners().map(p => ({ id: p.id, name: p.name })) — the frozen point-in-time option list (§9.3).

Partner-backed row rendering (§9.2–9.4)

  • buildEntryRow takes a { partnerBacked, partnerOptions } option. When partnerBacked: the identity-key field groups are not rendered; instead a single "Partner" dropdown group is rendered in their place (label "Partner", placeholder "— izberite —", options = partnerOptions with value=id, textContent=name). All non-identity fields render exactly as normal.
  • New builder in fieldBuilders.js: buildPartnerSelect(id, partnerOptions, value) (a select with the shared placeholder + one option per snapshot entry) + its {wrapper,input,error} trio via a small wrapper, so the existing .field/.field-error markup and aria-describedby wiring are reused.
  • On select (change): look up the chosen id in currentPartners() (live):
    • Exists → clear any stale flag; config.updateEntry(state, entry.id, config.partner.copyFrom(partner)) copies the partner's current data into the row's identity keys. From here the row owns an independent copy (§9.4).
    • Stale / removed (in the frozen snapshot but no longer in the live set) → set a persistent row-local staleSelected = true; clear the row's identity keys — config.updateEntry(state, entry.id, Object.fromEntries(config.partner.identityKeys.map(k => [k, '']))) — discarding whatever they previously held, rather than leaving them populated (§9.3, §9.4).
    • Placeholder re-selected (empty) → clear the stale flag and clear the identity keys the same way (§9.3).
    Mark the dropdown "touched", then refreshErrors().
  • Row message (single, at the dropdown, §9.3/§9.4): refreshErrors() for a partner-backed row computes config.validateEntry(entry) as usual but, instead of writing the identity keys' own error text (those inputs aren't rendered), resolves one dropdown message with this precedence:
    1. if staleSelected → "Partner ne obstaja." (persistent; independent of the Download gate; cleared only by selecting a currently-existing partner);
    2. else if the dropdown is touched/revealed and any identityKeys field has an error → "Izberite partnerja.";
    3. else → empty.
    Non-identity fields keep their own normal per-field error text.
  • Download gate is unchanged. Because the identity keys live in state, validateState/hasAnyErrors already catch a partner-backed KPR row whose supplier fields are blank (blocks) and already pass a partner-backed KIR row left blank (valid both-blank optional pair). No partner-specific carve-out; main.js's gate logic is untouched. revealErrors() marks the dropdown touched so the "Izberite partnerja." message surfaces on a blocked KPR row.
  • Stale check is selection-time only (§9.4): later deletion/edit of a partner does not re-run the check on an untouched row, so an already-copied row downloads fine with no indicator. No reactive subscription is needed for the indicator.

7. Key implementation concerns

  • Output correctness. The single most important invariant: a partner-backed row must serialize identically to a typed row with the same identity values. Guaranteed by copying into the existing identity keys and touching neither serialize.js nor the entry validators. Covered by an integration test asserting equal JSON for a typed vs. partner-backed row of the same data.
  • Validation vs. persistence failure (§6, §8, §11). Two distinct failure paths: a validation failure reveals inline errors and never reaches storage / never shows a bubble; a persistence failure (storage blocked) shows the action-specific error bubble, and the in-memory mutation is reverted so the list reflects reality.
  • Message precedence (§9.4). "Partner ne obstaja." (live, persistent, download-independent) outranks "Izberite partnerja." (download-gate driven). Hidden identity fields' own error text is never surfaced on a partner-backed row.
  • Frozen snapshot (§9.3). The dropdown option list is captured once at row creation and never re-synced; a new partner-backed row created later gets a fresh snapshot. Kept in the row closure, not in state.
  • Edge cases (§11). No partners → empty state + disabled add-partner buttons; cancel unsaved row → silent removal; second + with an unsaved row → scroll only; blank-required save → reveal, no persist; storage failure → error bubble + revert; duplicate name/VAT id → allowed (distinct ids); revert-to-saved edit → save disabled again; partner deleted after a row copied it → row downloads fine, no indicator; new partner-backed row after a deletion → fresh (post-deletion) snapshot; selecting a stale option or re-selecting the placeholder → row's identity keys cleared to empty, discarding any previously copied data (§9.3, §9.4).
  • Testing.
    • Unit (Node runner): partnerStore.test.js (load/save/round-trip, corrupt record, write failure via injected backend); state.test.js additions (add/update/remove, id sequencing incl. no in-session reuse + reload reuse, seeding from stored max); validate.test.js additions (validatePartner for each field, required in both flows).
    • Integration: typed-vs-partner-backed row serialize equality; partner-backed KPR blank blocks download while KIR blank does not.
    • Smoke (CDP): add/edit/delete + bubbles; dirty-gated save; single-unsaved-row rule; add-partner buttons enable after first partner across a tab switch; stale-option "Partner ne obstaja." persistence and identity-key clearing; placeholder re-selection clearing a previously-populated row; "Izberite partnerja." on a blocked KPR row; tab-switch state preservation; dark-mode bubble.

9. Assumptions & out of scope

  • Out of scope (spec §3): pagination, filtering, sorting, search; any change to DDV_KIR_KPR output/FURS fields/serialization/entry-validation rules; dedup/uniqueness enforcement; import/export, sync, or any server interaction.
  • Assumptions: the whole-partner-list write-through model is adequate (small list, no partial-write concerns); id reuse only after reload is acceptable (§3.3); the frozen dropdown snapshot and partner-backed flag live in the view row closure (no new state entry keys); tabs.js and the Download gate need no modification. All new user-facing text is the fixed Slovenian strings from the spec; code identifiers and internal errors stay English; FURS wire identifiers are untouched.
