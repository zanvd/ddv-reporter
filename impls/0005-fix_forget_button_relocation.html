Proportionality. This is a small, targeted layout fix in a single view module. The bug is a DOM-order problem, not a logic problem, so the plan introduces no new modules, state, dependencies, timers, CSS rules, or abstractions. Every change below traces to a requirement in spec 0005; nothing beyond it.

1. Overview

The persist-controls row (.persist-row) renders in src/js/view/persistControls.js as a flex row. The bug is purely DOM order: the save feedback span is appended between the two buttons, so showing/hiding it takes and releases horizontal space, shoving "Pozabi" sideways (spec §2). The fix collapses the two per-button feedback slots into one shared slot appended after both buttons. Because that slot now sits past both buttons, toggling its visibility can no longer move either button (spec R1/R2), and routing both actions through one slot gives last-action-wins for free (spec R3). No CSS, storage, validation, or copy changes.

2. Root cause

Current append order in renderPersistControls():

row.append(saveButton, saveFeedback.element, forgetButton, forgetFeedback.element);
//                      ^^^^^^^^^^^^^^^^^^^^^ interleaved between the buttons — the bug

Each feedback slot is a role="status" span, hidden by default (.persist-feedback[hidden] { display: none }) and shown for 3 s after its button's action. Since the save slot occupies flex space between the buttons, its show/hide shifts "Pozabi" horizontally.

4. The change, precisely

R2 — one shared feedback slot, appended after both buttons

createFeedbackSlot() already produces a self-contained role="status" span whose show(success, message) sets the success/error class, writes the text, unhides, and (re)arms a single hide timer. It is called once instead of twice. New append order: 

const sharedFeedback = createFeedbackSlot();
// ...
row.append(saveButton, forgetButton, sharedFeedback.element);
//         ^^^^^^^^^^^^^^^^^^^^^^^^^ buttons adjacent   ^^^^^^^ single shared area, last

R3 — last-action-wins, timer restart

Both handlers target the same slot:

saveButton.addEventListener('click', () => {
  const success = saveGeneral(state.general);
  sharedFeedback.show(success, success ? 'Shranjeno' : 'Poskusite znova.');
  if (success) refreshForgetEnabled();
});

forgetButton.addEventListener('click', () => {
  const success = forgetGeneral();
  sharedFeedback.show(success, success ? 'Pozabljeno' : 'Poskusite znova.');
  if (success) refreshForgetEnabled();
});
Last-action-wins is inherited, not added. show() already calls clearTimeout(hideTimer) and rewrites icon/text/class on every invocation. With a single shared slot, a save-or-forget click within the 3 s window immediately overwrites the visible message and restarts the timer — satisfying R3 and all §8 edge cases with no new timer or state logic (KISS/YAGNI). The existing FEEDBACK_DURATION_MS = 3000 constant is reused verbatim.

R1 — stable "Pozabi" position

With the shared slot appended after both buttons, its display: none ⇄ inline-flex transition only adds/removes space to the right of "Pozabi". Neither button moves. The reorder alone satisfies R1; per spec R1, no fixed-width reservation of the feedback area is introduced.

Doc comment

The module header comment (lines ~6–13) currently states "Each button's own transient feedback slot …". Update it to describe one shared, last-action-wins slot after both buttons, so the comment matches the code. Comment stays English per CLAUDE.md (developer-facing text).

5. Why no CSS change

Position is governed entirely by DOM order within the existing .persist-row flex container (display: flex; align-items: center; flex-wrap: wrap; gap: 0.75rem). The only thing changing is append order, in JS. The existing rules — .persist-feedback, .persist-feedback[hidden], and the success/error icon/text rules — apply unchanged to the single shared span (same class names). Adding CSS here would be unwarranted complexity.

Naming note. The saveFeedback / forgetFeedback local variable names collapse to a single sharedFeedback. These are internal identifiers only — no user-facing label is affected, and no FURS wire-format name is involved.

7. Testing

View modules have no unit tests in this repo (DOM-bound); logic modules do. Persist controls currently have no smoke coverage. Per the approved decision, this fix adds a lightweight smoke-test assertion via .claude/skills/run/ (serve src/, drive headless Chrome over CDP — see .claude/skills/run/smoke.mjs). Two invariants, matching the two things the fix changes:

  • R1 — position stability: read the "Pozabi" button's left offset (getBoundingClientRect().left) with no message shown, trigger a save so the shared message appears, and assert the offset is unchanged while the message is visible.
  • R3 — last-action-wins replacement: click save, then click forget within the window, and assert exactly one visible .persist-feedback whose text is the latest action's message ("Pozabljeno"), confirming replacement in the single shared slot.

Keep the assertion minimal and consistent with the existing smoke harness. No new unit-test files; no change to cd src && npm test logic-module suites (they are unaffected by a view-only change).
