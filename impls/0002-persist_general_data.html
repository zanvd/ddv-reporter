Scope: Technical/architecture only. The spec is the binding source of requirements; this plan describes how to build exactly what it specifies.

Settled decisions (this plan):
  1. Plain localStorage, accessed through one generic, DOM-free storage.js module. No dependency, no build step — consistent with the project's zero-dependency constraint.
  2. One key per data domain, namespaced ddvReporter.<domain> (Glava → ddvReporter.general); each value is a self-describing envelope { "version": 1, "data": {…} }. This is the answer to the spec §7 forward-compatibility constraint (§2 below).
  3. Two new non-view modules (storage.js generic, generalStore.js Glava-specific) and one new view module (view/persistControls.js). state.js, validate.js, and view/general.js stay unchanged.
  4. Auto-restore reuses the existing revealErrors() to satisfy the "restored fields are already touched" rule (§6.4), so general.js's quiet-until-touched flow is untouched.

1. Overview

Add opt-in, local-only persistence of the Glava section only to the existing fully client-side app. Two buttons — "Zapomni si" (save) and "Pozabi" (forget) — write/remove a snapshot of the six general-section fields to the browser's localStorage; on the next app open the form auto-restores from that snapshot. No backend, no network, nothing new leaves the browser. The feature introduces no tax logic and changes nothing about the DDV_KIR_KPR output, the download flow, or FURS validation (spec §10). Guiding principles: KISS / YAGNI — reuse the app's existing state, validation, and button visual language; add the smallest module set that satisfies the spec; and design the storage scheme so future persisted domains slot in beside Glava without migration (the one piece of deliberate, spec-mandated forward-looking structure — §2).

2. Storage design (central decision — spec §7)

The spec's binding constraint: Glava is the first of potentially several persisted data domains; the scheme must let later domains be added, read, and removed independently, with no migration or reshaping of what this feature stores, and not as a single flat blob under one key.

2.1 Key scheme — one key per domain

Each independent data domain gets its own localStorage key, under a fixed application namespace: 

ddvReporter.<domain>

  ddvReporter.general      ← this feature (the six Glava fields)
  ddvReporter.<future>     ← a later feature adds its own key, independently
  • Independently addressable: a future domain writes/reads its own key; it never reads or parses ddvReporter.general.
  • Independently removable: "Pozabi" calls removeItem("ddvReporter.general"), which touches only Glava. A future domain's forget touches only its own key.
  • No shared blob to migrate: because each domain is a separate key, adding a domain is purely additive — no existing record is restructured. This is the explicit rejection of the single-flat-blob approach the spec forbids.

2.2 Value shape — a self-describing envelope

Each key's value is a JSON string wrapping the domain payload in a small versioned envelope:

ddvReporter.general  =  {
  "version": 1,
  "data": {
    "taxPayerID": "12345678",
    "periodType": "monthly",
    "periodUnit": "6",
    "periodYear": "2026",
    "vracilo": false,
    "odbdelez": false
  }
}
  • Per-domain versioning lives inside the envelope (not in the key), so each domain can evolve its own shape later without a key rename and without affecting other domains' versions. Only version: 1 exists today; the read path is tolerant of anything it does not recognize (§6.4).
  • Payload = plain snapshot of the domain's fields — for Glava, exactly the six general-section state fields (§3), stored as-is (spec §6.1 "save-as-is, no validation gate").
Considered and rejected: (a) version in the key path (ddvReporter.v1.general) — pushes migration into key renaming and complicates "remove this domain"; the in-value envelope keeps the key stable and the version a per-domain concern. (b) A single ddvReporter key holding an object of all domains — this is exactly the flat blob the spec §7 forbids, since a later domain would have to read-modify-write (and risk clobbering) the shared record.

3. Domain model

No new in-memory entity is introduced — the persisted record is a snapshot of the existing state.general object (defined in plan 0001 §3.1, created by createState()). The persisted data payload is exactly the six Glava fields; KIR/KPR are never included (spec §3, §4).

Browser localStorage via getItem/setItem/removeItem Spec §1/§3 require single-browser, local-only, no-sync persistence that survives across sessions. localStorage is the exact fit: synchronous, per-origin, string key/value, no dependency. sessionStorage is wrong (does not survive a new session); IndexedDB is overkill for six scalar fields (YAGNI).
No new runtime dependency, no build step Project conventions (CLAUDE.md): vanilla ES-module JS, no framework/bundler. JSON (JSON.stringify/parse) covers envelope (de)serialization.

In some browsers even reading globalThis.localStorage or calling setItem throws; every access is guarded and reduced to a boolean success / null read.
Injectable storage backend (defaults to globalThis.localStorage) Keeps storage.js unit-testable under Node's built-in runner (no DOM/localStorage): tests pass a Map-backed fake and a throwing fake. Matches the project's DOM-free-pure-logic testing pattern.
Feedback icons as inline markup (Unicode/CSS), not image assets Spec §5/§6.6: reuse the existing status palette (--error, an app-consistent green), "no new visual language." Green check "✓" and red circle-with-X rendered from CSS tokens; no external assets (self-contained app).

6. Key implementation concerns

6.1 Saving — "Zapomni si" (spec §6.1)

  • Click → generalStore.saveGeneral(state.general) picks the six whitelisted fields and calls storage.writeRecord("general", 1, snapshot), overwriting any prior Glava record.
  • No validation gate — the snapshot is saved as-is regardless of validity/completeness. Validation is never invoked on the save path.
  • On success (true): show the green check next to "Zapomni si" for 3 s; enable "Pozabi".
  • On failure (false): show the red indicator + "Poskusite znova." next to "Zapomni si" for 3 s; write nothing; leave "Pozabi" state unchanged.

6.2 Forgetting — "Pozabi" (spec §6.2)

  • Click → generalStore.forgetGeneral() → storage.removeRecord("general"). Form values in memory are not altered.
  • On success: green check next to "Pozabi" for 3 s; disable "Pozabi".
  • On failure: red indicator + "Poskusite znova." next to "Pozabi" for 3 s; treat record as still present; keep "Pozabi" enabled.
  • Disabled state makes the button unreachable when nothing is saved; a stray call is a harmless no-op (spec §8).

6.3 "Pozabi" enabled/disabled (spec §6.3, §6.5)

  • Computed from generalStore.hasSavedGeneral() at mount, and updated after every successful save (→ enabled) / successful remove (→ disabled). A failed action does not change it.
  • Disabled = the native disabled attribute (functional) plus :disabled CSS (visual), satisfying "including visually."

6.4 Auto-restore + touched-on-load (spec §6.4, §8)

Bootstrap order in main.js (before the general view is built):

const state = createState();
const restored = loadGeneral();            // {…}|null, never throws
if (restored) updateGeneral(state, restored);   // overlay onto defaults
const generalView = renderGeneral(generalContainer, state);   // builds prefilled
…
if (restored) generalView.revealErrors();  // §6.4: restored fields already "touched"
  • Reusing revealErrors() is the whole trick: it marks the four validated general fields touched and re-runs validateGeneral, so any stale/invalid restored value surfaces its error immediately — without adding any parameter to renderGeneral or altering the quiet-until-touched path for subsequently typed input. When nothing is restored, revealErrors() is not called, so a fresh form stays quiet as today.
  • Tolerant read (generalStore.loadGeneral): overlay only recognized whitelist keys onto createState().general defaults; missing fields keep their default; unknown fields are ignored; any envelope version is accepted (only v1 exists) with the same whitelist filter. Corrupt/absent/unparseable → null → treated as "nothing saved," normal defaults, "Pozabi" disabled. Auto-restore never throws and never blocks startup.
  • KIR/KPR are untouched — they always start empty.

6.5 Feedback indicators & timers (spec §5, §6.5, §6.6)

  • Each button has its own feedback slot adjacent to it; only the clicked button's slot shows, reflecting the latest outcome.
  • Success: green check + accessible Slovenian text — "Shranjeno" (save) / "Pozabljeno" (remove). Failure: red circle-with-X + visible text "Poskusite znova."
  • Feedback element is a live region (role="status") with the Slovenian text as content so meaning is not color/icon-only; the icon glyph is aria-hidden.
  • 3 s transient: a per-controls setTimeout(…, 3000) hides the slot; a fresh click clearTimeouts the pending timer and restarts it for the latest outcome (spec §6.6 reset behavior). The failed state auto-returns to whichever prior state applies, since a failed action left storage unchanged (spec §6.5).

6.7 Testing

  • storage.js unit tests (Node runner, injected fake backend): namespacing (keyFor), envelope round-trip, tolerant read (missing key & corrupt JSON → null), and failure paths via a throwing fake (setItem/removeItem /property-access → false/null).
  • generalStore.js unit tests: whitelist filtering on save; defaults fallback for missing fields; ignore unknown fields; save/load round-trip; hasSavedGeneral true/false; forget clears.
  • Smoke test (.claude/skills/run/, headless Chrome over CDP) for the DOM/timer/localStorage paths the unit tests can't reach: save → "Pozabi" enables & green check appears; reload → form prefilled & restored-invalid errors shown immediately; forget → "Pozabi" disables. The existing integration.test.js (serialize pipeline) is unaffected — persistence is orthogonal to the JSON output and is not added there.

8. Assumptions and out-of-scope

Label to confirm at UI stage (not code-baked): the two button labels ("Zapomni si" / "Pozabi") and the failure text ("Poskusite znova.") are fixed by spec §5. Only the accessible success wording — proposed "Shranjeno" / "Pozabljeno" — is marked adjustable by spec §6.6; this plan carries them as the working values, to be confirmed rather than silently locked.

Assumptions

  • Target is current evergreen browsers with localStorage available in the normal case (failure modes are handled per §6.6, not assumed away).
  • Namespace token ddvReporter and domain name general are app-internal identifiers (not user-facing, so English per convention); future domains follow ddvReporter.<domain>.
  • The persist row mounts in a dedicated #general-persist div inside the general <section> and below #general-content, so general.js's replaceChildren() (scoped to #general-content) never disturbs it.
  • Stored values keep the same JS types they hold in state.general (strings and booleans), round-tripping cleanly through JSON; no numeric/type coercion beyond whitelist filtering is required.

Out of scope (restating spec §3 / §9 as it constrains this design)

  • Persisting KIR / KPR or anything beyond the six Glava fields.
  • Cross-device / cross-browser sync, cloud/account storage, any server component.
  • Encryption of stored data; multiple named/saved profiles; a save history.
  • Undo of "Pozabi"; export/import of the stored blob or any user-visible view of it.
  • Any change to the DDV_KIR_KPR output, the download gate, or FURS validation rules.
